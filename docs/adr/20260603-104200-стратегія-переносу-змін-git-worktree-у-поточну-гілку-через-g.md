## ADR Стратегія переносу змін git-worktree у поточну гілку через `getw`

## Context and Problem Statement

Команда `getw` переносила зміни з git-worktree у поточну гілку через `git checkout <branch> -- .`, що сліпо перезаписувало **усі** файли з worktree-гілки, затираючи файли, які змінювала лише поточна гілка. Окрім цього, `git apply --3way` падав з `does not match index` на брудному робочому дереві, де обидві гілки незалежно правили однакові файли. Потрібно було безпечно переносити лише дельту worktree-гілки без втрати змін поточної гілки.

## Considered Options

* `git checkout <branch> -- .` — сліпе перезаписування всіх файлів
* `git apply` (тільки дельта merge-base..target, без `--index`)
* `git apply --3way` — 3-way через індекс
* `git apply --reject` — неатомарний, зі збереженням `.rej` файлів
* Пофайловий `git merge-file --diff3` — 3-way без індексу, з конфліктними маркерами
* Багаторівневий пайплайн: `git apply` → `git merge-file --diff3` → mergiraf → LLM-агент

## Decision Outcome

Chosen option: "Багаторівневий пайплайн: `git apply` → `git merge-file --diff3` → mergiraf → LLM-агент", because кожен tier є дешевший за наступний: `git apply` для чистих патчів, пофайловий `git merge-file --diff3` обходить індекс і уникає `does not match index`, `mergiraf solve` (AST-рівень, опційний) зменшує навантаження на агента, LLM-агент лише для залишку маркерів; вердикт про успіх завжди залишається за скриптом (grep маркерів), а не делегується агенту.

### Consequences

* Good, because transcript фіксує очікувану користь: файли, змінені лише поточною гілкою, більше не затираються; `bun.lock` не мержиться рядково, а перегенеровується через `bun install` якщо `package.json` змінився у складному merge-шляху; при конфлікті worktree зберігається (fail-safe).
* Bad, because `git merge-file --diff3` позначає конфліктом навіть сусідні (не лише перетинні) зміни — такі хунки теж підуть на mergiraf/агента; mergiraf потребує встановлення (`brew install mergiraf`, автоматично в скрипті).
