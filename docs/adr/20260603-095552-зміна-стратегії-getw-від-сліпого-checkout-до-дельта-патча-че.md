## ADR Зміна стратегії getw: від сліпого checkout до дельта-патча через git apply

## Context and Problem Statement
Команда `getw` переносила зміни з git-worktree у поточну гілку за допомогою `git checkout "$target_branch" -- .`, що затирало усі файли робочого дерева версіями з worktree-гілки. Якщо поточна гілка мала власні зміни у файлах, яких worktree-гілка не торкалася, ці зміни беззворотно губилися без будь-яких попереджень.

## Considered Options
* `git checkout <branch> -- .` — сліпе перезаписування усього робочого дерева (попередня реалізація)
* `git diff --binary merge_base target | git apply` — перенесення лише дельти (merge-base..target) як unstaged-зміни

## Decision Outcome
Chosen option: "`git diff --binary merge_base target | git apply`", because це переносить лише файли, реально змінені у worktree-гілці, не зачіпаючи файли, змінені лише у поточній гілці.

### Consequences
* Good, because файли поточної гілки, яких worktree-гілка не чіпала, більше не затираються.
* Bad, because `git apply` без `--3way` / `--reject` падає атомарно при конфлікті — бодай один проблемний хунк блокує усе перенесення.

## More Information
Ключова зміна: `npm/getw.js`, рядок `git checkout "$target_branch" -- .` → `git diff --binary "$merge_base" "$target_branch" | git apply --whitespace=nowarn`. Changelog: `npm/.changes/`, bump `patch`, секція `Fixed`.

## ADR Обробка конфліктів у getw: від --3way до --reject + LLM-агент

## Context and Problem Statement
Перший варіант обробки конфліктів — `git apply --3way` — відмовляв з помилкою `does not match index`, якщо поточна гілка (`main`) мала uncommitted-зміни у тих самих файлах, що й worktree-гілка. Крім того, `git apply` є атомарним: один проблемний файл (наприклад, `bun.lock`) блокував накладання усіх решта хунків.

## Considered Options
* `git apply --3way` — 3-way merge через індекс; відмовляє при dirty working tree на спільних файлах
* `git apply --reject` + LLM-агент для `.rej`-файлів
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`git apply --reject` + LLM-агент", because `--reject` не торкається індексу (немає `does not match index`), є неатомарним (придатні хунки лягають, проблемні осідають у `*.rej`), а агент (`claude -p`, фолбек `cursor-agent -p`) інтегрує лише ці `.rej` і видаляє їх; критерій успіху — відсутність `*.rej` після агента.

### Consequences
* Good, because transcript фіксує очікувану користь: знято блокування на dirty working tree, `bun.lock` більше не блокує решту файлів, fail-safe — worktree зберігається якщо `.rej` лишилися.
* Bad, because для видалення `.rej` агенту потрібен дозвіл `Bash(rm:*)` (`--allowedTools "Edit,Write,MultiEdit,Read,Bash(rm:*)"`); якщо у конкретному середовищі `rm` заблоковано, агент не може прибрати `.rej` і `getw` безпечно зберігає worktree з повідомленням про ручне доведення.

## More Information
Реалізація: `npm/getw.js` — функція `_getw_resolve_with_agent()` з prompt-ом для агента; блок `git apply --reject --whitespace=nowarn "$patch_file"` → `_getw_list_rejects` → `_getw_resolve_with_agent` → перевірка залишкових `.rej`. `claude -p` викликається з `--permission-mode acceptEdits --allowedTools "Edit,Write,MultiEdit,Read,Bash(rm:*)" --model "${GETW_MERGE_MODEL:-sonnet}"`, фолбек — `cursor-agent -p --force --output-format text`. Changelog: `npm/.changes/`, bump `patch`, секція `Fixed`.
