## `ch.js` vs `runChangeCli` (канон)

| Аспект | `@7n/n ch` | `n-cursor change` |
|---|---|---|
| `--bump` | опціональний → дефолт `minor` | **обов'язковий** |
| `--section` | опціональний → дефолт `Changed` | **обов'язковий** |
| `--message` | обов'язковий | обов'язковий |
| `--ws` | опціональний → `.` | опціональний → `.` |
| Валідація bump/section | делегується каноном (через `parseChangeFile`) | власна, теж через `parseChangeFile` |
| Error при invalid bump | канон відповідає своїм `❌` | те саме |

## Висновок

Збігаються повністю — **крім навмисних дефолтів** (`minor`/`Changed`). `ch.js` — строгий надмножина: будь-який виклик `n-cursor change` можна записати через `@7n/n ch` з тими ж флагами, і навпаки. Ненавмисного дрейфу нема.

Одна деталь: `ch.js` перевіряє лише «`--message` відсутній» перед делегацією, а валідацію значень (`bump`/`section` ∈ допустимих) виконує канон. Тобто `npx @7n/n ch --bump TYPO --message x` поверне помилку з `@nitra/cursor`, а не з `@7n/n`. Це нормально (поведінка правильна), але повідомлення прийде не в стилі `@7n/n`.
