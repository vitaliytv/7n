## ADR Перенесення лише дельти worktree-гілки замість сліпого `git checkout -- .`

## Context and Problem Statement
Команда `getw` переносила зміни з worktree-гілки у поточну гілку через `git checkout "$target_branch" -- .`, що замінювало **всі** файли робочого дерева вмістом worktree-гілки. Якщо поточна гілка мала зміни у файлах, яких worktree не чіпав, ці зміни затирались мовчазно без виявлення конфліктів.

## Considered Options
* `git checkout <branch> -- .` (поточна реалізація — сліпе перезаписування)
* `git diff <merge-base> <target> | git apply` (перенесення лише дельти від спільного предка)

## Decision Outcome
Chosen option: "`git diff --binary <merge-base> <target> | git apply`", because це переносить лише зміни, зроблені у worktree-гілці відносно спільного предка, і не чіпає файли, яких ця гілка не змінювала, зберігаючи локальні правки у поточній гілці.

### Consequences
* Good, because файли, змінені лише в поточній гілці (`main`), більше не затираються.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/getw.js`. Змінено рядок `git checkout "$target_branch" -- .` на `git diff --binary "$merge_base" "$target_branch" | git apply --whitespace=nowarn`. Для визначення merge-base використовується `git merge-base "$current_branch" "$target_branch"`.

## ADR Багаторівневий резолв конфліктів у `getw` через `git merge-file`, mergiraf і LLM-агент

## Context and Problem Statement
Після переходу на delta-transfer через `git apply`, конфліктні зміни (коли обидві гілки правили той самий файл) потребували окремої стратегії. Спроби `git apply --3way` провалювались з помилкою `does not match index` (бо worktree-гілки незалежно змінювали ті самі tooling-файли), а `git apply --reject` ускладнював визначення успіху через делегування видалення `.rej` самому агенту.

## Considered Options
* `git apply --3way` (конфліктні маркери через індекс)
* `git apply --reject` + агент видаляє `.rej`
* Пофайловий `git merge-file --diff3` (без індексу) + mergiraf (AST, Tier 2) + LLM-агент (Tier 3), скрипт визначає вердикт

## Decision Outcome
Chosen option: "пофайловий `git merge-file --diff3` + mergiraf + LLM-агент з вердиктом у скрипта", because `git merge-file` працює лише по файлах без індексу (немає `does not match index`), скрипт детерміновано перевіряє наявність маркерів (`<<<<<<<`, `>>>>>>>`) після кожного tier'у і лише при їх залишку передає список агенту; агент робить лише творчу частину (прибирає маркери), не вирішує що вважати успіхом.

### Consequences
* Good, because скрипт фіксує вердикт: worktree зберігається лише коли після агента лишились маркери — дані не втрачаються мовчки.
* Good, because transcript фіксує очікувану користь: mergiraf (Tier 2) знімає хибно-позначені конфлікти через AST-аналіз, зменшуючи навантаження на агента.
* Good, because `--diff3` зберігає base-секцію `|||||||`, яку потребує mergiraf.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/getw.js`, `npm/README.md`. Helpers: `_getw_files_with_markers`, `_getw_ensure_mergiraf` (auto-install: `brew install mergiraf`, фолбек `cargo install --locked mergiraf`), `_getw_mergiraf_solve`, `_getw_resolve_with_agent`. Env: `GETW_NO_MERGIRAF=1` — вимкнути Tier 2; `GETW_MERGE_MODEL`, `GETW_MERGE_CURSOR_MODEL` — моделі агентів. LLM: `claude -p --permission-mode acceptEdits --allowedTools "Edit,Write,MultiEdit,Read"`, фолбек `cursor-agent -p`. Функціональний тест tier'ів проводився на тимчасовому репозиторії з fake-mergiraf.

## ADR Спеціальна обробка `bun.lock`: пропустити merge, запустити `bun install` лише при розбіжності

## Context and Problem Statement
`bun.lock` є lock-файлом, зміни в якому генеруються інструментом (`bun install`), а не правляться вручну. Пофайловий merge через `git merge-file` для lock-файлів дає лише шум (детерміністичний формат, але не рядковий). Потрібно було вирішити, коли та як оновлювати `bun.lock` після переносу змін.

## Considered Options
* Мержити `bun.lock` як звичайний файл (через merge-file/mergiraf/агент)
* Брати версію з target-гілки завжди
* Пропускати merge, запускати `bun install` якщо lock-файли відрізняються

## Decision Outcome
Chosen option: "пропускати merge `bun.lock`, запускати `bun install` лише коли файл у поточній гілці і у target реально відрізняються", because якщо lock-файли однакові — `bun install` не потрібен; якщо різні — `bun install` перегенерує lock відповідно до вже змердженого `package.json`.

### Consequences
* Good, because `bun install` не запускається без потреби (lock-файли збігаються — операція пропускається).
* Good, because transcript фіксує очікувану користь: lock перегенерується інструментом, а не зливається рядково — результат детермінований і коректний.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/getw.js`. Helper `_getw_bun_lock_differs` порівнює вміст через `cmp -s`: ours береться з робочого дерева або з `HEAD` поточної гілки, theirs — через `git show "$tgt:$rel"`. Аналогічно обробляються `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock` (ці беруться з target без `bun install`). Прапорець `regen_bun=1` та змінна `regen_bun_path` виставляються в циклі merge; `bun install` запускається лише після повністю успішного мержу (без залишку маркерів).
