## ADR getw: від сліпого перезаписування до багаторівневого мержу

## Context and Problem Statement
Команда `getw` для перенесення змін з git-worktree у поточну гілку використовувала `git checkout "$target_branch" -- .`, яка **сліпо перезаписувала** всі файли робочого дерева вмістом worktree-гілки. Файли, змінені лише в поточній гілці (але незмінені у worktree-гілці), затиралися їхніми старими версіями без будь-яких попереджень чи виявлення конфліктів.

## Considered Options
* Продовжувати використовувати `git checkout <branch> -- .` (поточний стан)
* Переносити лише дельту (`git diff merge-base..target | git apply`) без конфліктної логіки
* Пофайловий 3-way через `git merge-file` + багаторівневий резолв (обраний варіант)
* `git apply --3way` (відхилено: падає з `does not match index` на dirty робочому дереві)
* `git apply --reject` з делегуванням видалення `.rej`-файлів агенту (відхилено: вердикт про успіх не можна делегувати агенту)

## Decision Outcome
Chosen option: "пофайловий `git merge-file --diff3` + багаторівневий авторезолв", because `git apply --3way` падав з `does not match index` коли індекс поточної гілки не збігався з preimage-блобами патча (обидві гілки незалежно правили ті самі файли), а `git apply --reject` + делегування видалення `.rej` агенту давало крихкий вердикт (залежав від дозволу на `rm`). `git merge-file` працює лише по файлах без індексу, скрипт детерміновано виносить вердикт через grep маркерів, агент лише прибирає маркери.

### Consequences
* Good, because файли, змінені лише в поточній гілці, більше не затираються (перевірено функціональним тестом усіх кейсів: `ours_only.txt` збережено).
* Good, because розподіл ролей чіткий: скрипт (детерміновано) ставить конфліктні маркери, авторезолвери намагаються їх прибрати, агент — лише творча частина на залишку; вердикт завжди за скриптом.
* Good, because багаторівневий підхід мінімізує кількість звернень до LLM-агента: Tier 0 (`git apply`), Tier 1 (`git merge-file --diff3`), Tier 2 (`mergiraf solve`, опційно), Tier 3 (LLM).
* Good, because lock-файли (`bun.lock`, `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`) беруться детерміновано з версії target, оминаючи безглуздий line-based merge.
* Bad, because `git merge-file --diff3` маркує конфліктом навіть сусідні (не лише перетинні) зміни — трохи більше шуму для агента, ніж теоретично необхідно.
* Bad, because Tier 2 (mergiraf) вимагає окремої установки (`cargo install --locked mergiraf`) і за відсутності мовчки пропускається.

## More Information
* Змінені файли: `npm/getw.js`, `npm/README.md`
* Функціональний тест merge-логіки: `/tmp/getw_func/run.sh` (видалено після перевірки); тест Tier-2 wiring з fake-mergiraf: `/tmp/getw_t2/run.sh`
* Ключові функції у `npm/getw.js`: `_getw_resolve_with_agent`, `_getw_files_with_markers`, `_getw_mergiraf_solve`
* `mergiraf solve <file>` — розв'язує конфлікти in-place у файлі з diff3-маркерами; exit 0 = все розв'язано, 1 = не розпарсив, 2 = лишились конфлікти; невідомі типи лишають маркери і безпечно падають на агента
* Змінні оточення: `GETW_MERGE_MODEL` (модель для claude), `GETW_MERGE_CURSOR_MODEL` (модель для cursor-agent), `GETW_NO_MERGIRAF=1` (вимкнути mergiraf)
