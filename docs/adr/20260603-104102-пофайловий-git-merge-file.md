## ADR пофайловий git merge-file

## Context and Problem Statement
Команда `getw` переносить зміни з git-worktree у поточну гілку. Початкова реалізація використовувала `git checkout <branch> -- .` — «сліпе перезаписування», яке затирало файли, змінені тільки в поточній гілці, без виявлення конфліктів.

## Considered Options
* `git checkout <branch> -- .` (поточна реалізація — сліпе перезаписування)
* `git apply` з дельтою `merge-base..target` (без `--index`, незастейджено, атомарний)
* Пофайловий `git merge-file --diff3` (без індексу, неатомарний, з маркерами)

## Decision Outcome
Chosen option: "Пофайловий `git merge-file --diff3`", because він працює лише по файлах (без індексу), не виникає помилка `does not match index` навіть при брудному робочому дереві, неатомарний (один невдалий файл не блокує решту), і залишає diff3-маркери `|||||||` у потрібному форматі для mergiraf (Tier 2 авторезолвера).

### Consequences
* Good, because файли, змінені тільки в поточній гілці, більше не затираються — зберігається детермінований 3-way merge по кожному файлу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/getw.js`. Команда генерує zsh-скрипт (`ZSH_SCRIPT`) з функцією `getw()`. Ключові зміни: `git diff --binary merge_base target | git apply --whitespace=nowarn` → пофайловий `git merge-file -p -L ... --diff3`. Функціональний тест підтвердив коректну обробку: «тільки theirs», «тільки ours», «обидва без перетину», «обидва з конфліктом», «додано в theirs», «видалено в theirs», «бінарний» (`exit 255` → версія target).

## ADR Багаторівневий резолв конфліктів у getw

## Context and Problem Statement
Після переходу на `git merge-file` конфлікти залишали маркери у файлах. Потрібна стратегія розв'язання, яка мінімізує участь LLM-агента, надаючи перевагу детермінованим авторезолверам.

## Considered Options
* Одразу делегувати агенту (`claude -p`) всі конфлікти
* Спочатку `git apply --reject` + агент розв'язує `.rej`-файли й видаляє їх
* Спочатку `git apply --reject` + агент розв'язує `.rej`-файли + скрипт видаляє їх
* Багаторівневий: `git merge-file --diff3` → `mergiraf solve` → LLM-агент (скрипт виносить вердикт по маркерах)

## Decision Outcome
Chosen option: "Багаторівневий: `git merge-file --diff3` → `mergiraf solve` → LLM-агент", because скрипт повністю контролює детерміновану частину (постановка маркерів, вердикт по наявності `<<<<<<<`/`>>>>>>>`), агент робить лише творчу роботу (прибирає маркери), а mergiraf структурно розв'язує конфлікти AST-рівня до залучення LLM.

### Consequences
* Good, because transcript фіксує очікувану користь: mergiraf знімає помилкові line-based конфлікти, агент бачить менше роботи; вердикт скрипта детермінований — жодної прихованої втрати даних.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `npm/getw.js`, `npm/README.md`. Helper-функції: `_getw_ensure_mergiraf` (авто-встановлення через `brew install mergiraf`, фолбек `cargo install --locked mergiraf`), `_getw_mergiraf_solve`, `_getw_files_with_markers`, `_getw_resolve_with_agent`. Env-змінні: `GETW_NO_MERGIRAF=1`, `GETW_MERGE_MODEL` (default `sonnet`), `GETW_MERGE_CURSOR_MODEL`. Агент отримує `--permission-mode acceptEdits --allowedTools "Edit,Write,MultiEdit,Read"` — без `Bash(rm:*)` (видаленням `.rej` керує скрипт, а не агент). Функціональні тести: `zsh -n` синтаксис, реальний `git merge-file` у тимчасовому репо з fake-mergiraf.

## ADR Окрема обробка bun.lock у getw

## Context and Problem Statement
`bun.lock` є бінарним-по-суті lockfile, злиття якого через `git merge-file` породжує лише шум. Водночас після успішного merge `package.json` може змінитись, і lockfile потребує перегенерації.

## Considered Options
* Брати версію target (як інші lock-файли: `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`)
* Відкласти `bun.lock`, не мержити його, запускати `bun install` після успішного мержу

## Decision Outcome
Chosen option: "Відкласти `bun.lock`, запускати `bun install` після успішного мержу", because `bun install` правильно звіряє lock зі змердженим `package.json` і не залишає конфліктних артефактів.

### Consequences
* Good, because `bun.lock` завжди консистентний з `package.json` після мержу.
* Bad, because `bun install` потребує мережі або локального кешу; якщо `bun` відсутній у `PATH` — виводиться попередження без автоматичного виправлення.

## More Information
Файл: `npm/getw.js`. Умова запуску `bun install`: `bun.lock` або `package.json` присутні в дельті `merge-base..target` незалежно від шляху мержу (чистий `git apply` або пофайловий). Lock-файли з іншими рушіями (`package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`) продовжують братися з target без перегенерації. Функціональний тест: fake-bun підтвердив відкладення та виклик `bun install` наприкінці.
