## ADR Перенесення лише дельти worktree-гілки замість сліпого `git checkout -- .`

## Context and Problem Statement
`getw` переносив файли з worktree-гілки командою `git checkout "$target_branch" -- .`, яка перезаписувала **всі** файли поточної гілки вмістом worktree-гілки, не виявляючи конфліктів. Якщо в `main` були зміни у файлах, яких worktree-гілка не чіпала, ці зміни затиралися старими версіями з worktree без будь-якого попередження.

## Considered Options
* Зберегти `git checkout <branch> -- .` (сліпе перезаписування)
* Переносити лише дельту через `git diff merge-base..target | git apply`

## Decision Outcome
Chosen option: "Переносити лише дельту через `git diff merge-base..target | git apply`", because `git apply` без `--index` кладе зміни як unstaged, не чіпає файли поза патчем, а при конфлікті зупиняється замість мовчазної втрати даних.

### Consequences
* Good, because файли, змінені лише в поточній гілці, більше не затираються версіями з worktree-гілки.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/getw.js`. Ключова зміна: `git checkout "$target_branch" -- .` → `git diff --binary "$merge_base" "$target_branch" | git apply --whitespace=nowarn`. Спільний предок знаходиться через `git merge-base "$current_branch" "$target_branch"`.

## ADR Пофайловий `git merge-file` замість `git apply --3way` / `--reject` для конфліктного мержу

## Context and Problem Statement
Перший варіант конфліктного резолву (`git apply --3way`) падав з помилкою `does not match index`, бо обидві гілки незалежно правили одні й ті самі tooling-файли, і індекс `main` не збігався з preimage-блобами патча. Другий варіант (`git apply --reject`) потребував, щоб агент видаляв `.rej`-файли, що робило вердикт про успіх залежним від прав агента на `rm` та від того, чи агент взагалі видалить файли.

## Considered Options
* `git apply --3way` (через індекс, з конфліктними маркерами)
* `git apply --reject` (без індексу, залишки у `*.rej`, агент видаляє `.rej`)
* Пофайловий `git merge-file --diff3` (лише по файлах, без індексу; скрипт ставить маркери, агент лише прибирає їх, скрипт виносить вердикт)

## Decision Outcome
Chosen option: "Пофайловий `git merge-file --diff3`", because він працює лише по файлах без звернення до індексу (усуває `does not match index` в принципі), а вердикт про успіх (grep маркерів `<<<<<<<`/`>>>>>>>`) залишається детермінованим і повністю за скриптом — агент робить лише творчу частину.

### Consequences
* Good, because розподіл ролей чіткий: скрипт детерміновано ставить маркери і перевіряє результат, агент лише прибирає маркери з наявних файлів.
* Good, because `--diff3` лишає base-секцію `|||||||`, яка потрібна Mergiraf для AST-резолву.
* Bad, because `git merge-file` може маркувати конфлікт навіть для сусідніх (не лише перетинних) змін — це стандартна поведінка diff3.

## More Information
Файл: `npm/getw.js`. Helper `_getw_files_with_markers` grep-ає `^(<<<<<<<|>>>>>>>)` для детермінованого вердикту. Функціональний тест перевіряв кейси: `theirs only`, `ours only` (зберігається), неперетинні зміни (зливаються без маркерів), справжній конфлікт (маркери), додавання/видалення, бінарні файли.

## ADR Багаторівневий пайплайн резолву конфліктів: merge-file → mergiraf → LLM-агент

## Context and Problem Statement
Після впровадження `git merge-file` залишились конфлікти, які line-based diff3 маркував дарма (зміни в різних частинах синтаксичного дерева). Відправляти їх відразу на LLM-агента — дорого; потрібні дешевші детерміновані авторезолвери, які «закривають» прості випадки перед агентом.

## Considered Options
* Лише LLM-агент для всіх конфліктів
* `git merge-file` → LLM-агент (без проміжного авторезолвера)
* `git merge-file --diff3` → Mergiraf (AST/tree-sitter) → LLM-агент

## Decision Outcome
Chosen option: "`git merge-file --diff3` → Mergiraf → LLM-агент", because кожен рівень закриває свій клас конфліктів найдешевшим інструментом: merge-file зливає неперетинні зміни; mergiraf (Rust, tree-sitter, 25+ мов) розв'язує конфлікти у різних синтаксичних вузлах; LLM-агент отримує лише реальний залишок.

### Consequences
* Good, because transcript фіксує очікувану користь: mergiraf розв'язує конфлікти, які line-based diff3 позначив дарма, не пускаючи їх до агента.
* Good, because вердикт завжди за скриптом (grep маркерів), незалежно від того, який tier відпрацював.
* Bad, because mergiraf потребує встановлення; для невідомих типів (`.mdc`, lock-файли) він лишає маркери — ці кейси все одно йдуть до агента.

## More Information
Файл: `npm/getw.js`. Tier 0: чистий `git apply`. Tier 1: `git merge-file --diff3` (без індексу). Tier 2: `mergiraf solve <file>` (off через `GETW_NO_MERGIRAF=1`). Tier 3: `claude -p --permission-mode acceptEdits --allowedTools "Edit,Write,MultiEdit,Read"`, фолбек `cursor-agent -p`. Якщо `mergiraf` відсутній — `_getw_ensure_mergiraf` ставить його через `brew install mergiraf`, фолбек `cargo install --locked mergiraf`. Тест з fake-mergiraf підтвердив коректний wiring tier'ів.

## ADR Окрема обробка `bun.lock`: пропуск мержу + умовний `bun install`

## Context and Problem Statement
`bun.lock` — автоматично згенерований файл; пофайловий merge-file дає по ньому лише шум (обидві сторони незалежно регенерували lock), а відправляти його Mergiraf чи агенту немає сенсу. При цьому запускати `bun install` після кожного `getw` теж зайве, якщо lock-файли збіглися.

## Considered Options
* Мержити `bun.lock` як звичайний файл (через merge-file/mergiraf/агент)
* Брати версію `bun.lock` з target (worktree-гілки) без перегенерації
* Пропустити `bun.lock` у циклі мержу; запускати `bun install` лише коли `ours` і `theirs` відрізняються

## Decision Outcome
Chosen option: "Пропустити `bun.lock` у циклі мержу; запускати `bun install` лише коли `ours` і `theirs` відрізняються", because lock — результат `bun install`, його «правильна» версія є продуктом виконання `bun install` над змердженим `package.json`, а не результатом text-merge двох lock-файлів.

### Consequences
* Good, because `bun install` не запускається без потреби, якщо обидва lock-файли вже збігаються.
* Good, because складний конфліктний merge автоматично згенерованого файлу виключений.
* Bad, because якщо `package.json` змінився, а `bun.lock` у дельті відсутній — `bun install` не запуститься автоматично (хоча зазвичай зміна залежностей тягне і зміну lock-файлу).

## More Information
Файл: `npm/getw.js`. Helper `_getw_bun_lock_differs` порівнює `ours` (робоче дерево або `HEAD`) і `theirs` (blob з `target_branch`) через `cmp -s`. Змінна `regen_bun_path` зберігає шлях. `bun install` запускається лише після повністю успішного мержу (без маркерів), бо на конфліктах раніше — `return 1`. Аналогічно обробляються `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock` (беруться з target без перегенерації).
