## ADR Delta-мердж замість повного `git checkout` зрізу в `getpull`

## Context and Problem Statement
Перша реалізація `getpull` робила `git checkout origin/$branch -- .` — повний checkout усіх файлів з origin, що затирало незакомічені правки tracked-файлів у поточному робочому дереві. Проблема виявлена при реальному запуску: команда відпрацювала безпечно лише тому, що origin випадково був попереду.

## Considered Options
* Переносити лише дельту `merge-base(HEAD, origin/<branch>)..origin/<branch>` через `git apply` + tier-пайплайн — так само, як у `getw`
* Залишити повний `git checkout origin/$branch -- .` (швидко, але деструктивно)

## Decision Outcome
Chosen option: "Переносити лише дельту через спільне ядро `_n7merge_delta`", because `git checkout` зрізу затирає будь-які незакомічені правки tracked-файлів; delta-підхід торкається лише файлів, що реально змінились між `merge-base` і `origin/<branch>`.

### Consequences
* Good, because transcript фіксує очікувану користь: емпірична перевірка підтвердила, що staged, unstaged і локальні коміти (не з origin) лишаються незайманими після `getpull`.
* Bad, because delta-підхід при текстовому конфлікті лишає маркери у робочому дереві (суто-unstaged оригінальний вміст конфліктного файлу живе лише всередині маркерів і не має окремого бекапу без pre-flight знімка).

## More Information
Команди перевірки в transcript: `git diff --stat origin/main` (порожньо = WT ≡ origin/main); `git diff --cached` (staged ціле); `git rev-parse HEAD` (незмінний). Функція `_n7merge_delta(ours, src)` обчислює `base = git merge-base "$ours" "$src"` і патч `git diff "$base" "$src"`, після чого `git apply --whitespace=nowarn`.

## ADR Pre-flight бекап через `git stash create`/`store` замість `git stash push`

## Context and Problem Statement
Delta-мердж безпечний для staged і закомічених змін, але суто-unstaged вміст конфліктного файлу після тривимірного diff3 живе лише всередині маркерів — окремої копії немає. Потрібна точка відкату, яка не порушує tier-пайплайн і не змінює стан дерева.

## Considered Options
* `git stash create` + `git stash store` — знімок-commit без очищення дерева (дерево не чіпається)
* `git stash push` — повний stash з очищенням дерева
* Без бекапу (попередній стан)

## Decision Outcome
Chosen option: "`git stash create` + `git stash store`", because на відміну від `git stash push`: не очищає робоче дерево, не порушує staged/unstaged-розрізнення, не переносить конфлікт на `stash pop` повз tier-пайплайн, не створює проблем на чистому дереві (порожній вивід `stash create` → крок пропускається автоматично).

### Consequences
* Good, because transcript фіксує очікувану користь: емпірична перевірка (`/tmp/gpbak3`) підтвердила, що знімок реально містить і staged, і unstaged правки (`git show <sha>:a.txt` → `UNSTAGED_TOP`, `git show <sha>:d.txt` → `STAGED_D`); на чистому дереві крок пропускається без зайвого виводу.
* Bad, because знімок лишається в `git stash list` після успішного мерджу і потребує ручного `git stash drop` — автоматичне очищення після успіху не реалізовано (transcript підтверджує: інструкція `git stash drop` друкується в stdout, але не виконується автоматично).

## More Information
Реалізовано на початку `_n7merge_delta` у `npm/merge.js`. У stdout друкується `🛟 Бекап … збережено: git stash apply <sha> (відновити) · git stash drop (прибрати)`. Тест: `npm/tests/merge.test.mjs` перевіряє виклики `git stash create` і `git stash store` через mocked `spawnFn`. Команди перевірки в transcript: `git stash list` (до/після), `git show <sha>:a.txt`.

## ADR Per-file підсумок Tier-3-агента в stdout без файлового артефакту

## Context and Problem Statement
При досягненні Tier 3 (LLM-агент прибирає конфліктні маркери) єдиним зворотним зв'язком був детермінований лог скрипта (які файли конфліктні + exit code). Не було структурованого пояснення, що саме і як розв'язав агент, — знання губилось після закриття терміналу.

## Considered Options
* Вимагати від агента per-file підсумок у відповіді (stdout), без збереження у файл
* Зберігати підсумок у файл-артефакт (`.getmerge.log`)
* Залишити лише детермінований лог скрипта (попередній стан)

## Decision Outcome
Chosen option: "Per-file підсумок у stdout без файлового артефакту", because артефакт явно відхилив користувач («без артефакту»); zsh запускається зі `stdio:'inherit'`, тому відповідь агента наживо видна в терміналі.

### Consequences
* Good, because підсумок (що хотіла кожна сторона і як агент примирив конфлікт) іде наживо в термінал без зайвих файлів у репозиторії.
* Bad, because підсумок існує лише в буфері терміналу — після закриття сесії він недоступний; якість і деталізація залежать від моделі (контракту нема, лише промпт-інструкція).

## More Information
Додано до промпта в `_n7merge_resolve_with_agent` у `npm/merge.js`: «Наприкінці надрукуй (у відповіді, НЕ у файли) короткий підсумок по КОЖНОМУ файлу: 1-2 рядки — що хотіла кожна сторона у конфлікті і як саме ти це примирив.» Тест у `npm/tests/merge.test.mjs` перевіряє наявність цієї інструкції в рядку промпту через `expect(prompt).toContain(...)`.
