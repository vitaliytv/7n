# getw: Пофайловий `git merge-file --diff3` та багаторівневий резолв конфліктів

**Status:** Accepted
**Date:** 2026-06-03

## Context and Problem Statement

Команда `getw` в `npm/getw.js` переносила зміни з git-worktree у поточну гілку через `git checkout <branch> -- .`, що сліпо перезаписувало робоче дерево цілком — файли, змінені лише в поточній гілці, затиралися старою версією з worktree-гілки без виявлення конфліктів. Після переходу на `git apply` виникала нова проблема: при двосторонніх конфліктах (обидві гілки правили однакові tooling-файли) `git apply --3way` падав із `does not match index`, а `git apply --reject` потребував делегування видалення `.rej`-файлів агенту, що робило вердикт про успіх залежним від дозволів та поведінки агента.

## Considered Options

* `git checkout <branch> -- .` — оригінальний підхід (сліпа заміна всього дерева)
* `git apply` (чиста дельта merge-base..target, без `--index`)
* `git apply --3way` — тризначний мерж через індекс; падає з `does not match index` на dirty робочому дереві
* `git apply --reject` — без індексу, але делегує вердикт агенту (крихко)
* Пофайловий `git merge-file --diff3` + tiered авторезолвери (обраний варіант)

## Decision Outcome

Chosen option: "Пофайловий `git merge-file --diff3` з багаторівневим резолвом конфліктів", because це єдиний підхід, що одночасно: (1) уникає `does not match index` (працює лише по файлах, без індексу); (2) дає детермінований вердикт скрипта (grep маркерів `<<<<<<<`/`>>>>>>>`); (3) чітко розподіляє ролі — скрипт ставить маркери й виносить вердикт, агент лише прибирає маркери; (4) дозволяє дешеві детерміновані tier'и перед дорогим LLM-агентом.

Рівні резолву (від найдешевшого):
- **Tier 0** — `git apply` (чистий патч дельти; якщо успішно — подальші tier'и не запускаються)
- **Tier 1** — `git merge-file --diff3` per-file (без індексу; `--diff3` залишає `|||||||` base-секцію для mergiraf)
- **Tier 2** — `mergiraf solve` (AST/tree-sitter авторезолвер; авто-встановлюється через `brew install mergiraf`, фолбек `cargo install --locked mergiraf`; вимикається `GETW_NO_MERGIRAF=1`)
- **Tier 3** — LLM-агент (`claude -p --permission-mode acceptEdits`, фолбек `cursor-agent -p`) лише на залишок з маркерами

`bun.lock` обробляється окремо: виключається з merge-циклу (`regen_bun=1`) і перегенеровується через `bun install` після успішного мержу — тригер: `bun.lock` або `package.json` присутні в дельті `merge-base..target` (незалежно від шляху мержу). Інші lockfile (`package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`) беруться детерміновано з версії target.

### Consequences

* Good, because файли, змінені лише в поточній гілці, більше не затираються — `git merge-file` зберігає «ours» там, де «theirs» не міняло ці рядки (підтверджено функціональним тестом кейсу «тільки ours змінив»).
* Good, because вердикт про успіх мержу — детермінований grep маркерів у скрипті (`_getw_files_with_markers`), не залежить від поведінки агента; агент отримує лише права `Edit,Write,MultiEdit,Read` — без `Bash(rm:*)`.
* Good, because `bun.lock` завжди фінально консистентний зі `package.json` після мержу.
* Good, because Tier 2 (mergiraf) є опційним, авто-встановлюється і підтримує 25+ мов на AST-рівні, знімаючи хибні line-based конфлікти до виклику LLM.
* Bad, because `git merge-file` може генерувати конфліктні маркери навіть для сусідніх (не лише перетинних) змін — такі false-conflict передаються Tier-2/3 замість автоматичного зливання.
* Bad, because Tier 2 (mergiraf) вимагає окремої інсталяції (`brew install mergiraf` або `cargo install --locked mergiraf`), яка може зайняти кілька хвилин; за відсутності — Tier 2 мовчки пропускається.
* Bad, because `bun install` потребує мережевого доступу або локального кешу; якщо `bun` відсутній — lockfile лишається в стані local-ours із попередженням.

## More Information

* Змінені файли: `npm/getw.js`, `npm/README.md`
* Ключові функції у `npm/getw.js`: `_getw_ensure_mergiraf`, `_getw_mergiraf_solve`, `_getw_files_with_markers`, `_getw_resolve_with_agent`; merge-цикл (рядки ~130–270)
* Tier 0: `git diff --binary "$merge_base" "$target_branch" | git apply --whitespace=nowarn`
* Tier 1: `git merge-file -p -L ... --diff3 ours base theirs > merged`
* `mergiraf solve <файл>` — in-place, вимагає diff3-маркерів із `|||||||` base-секцією; exit 0 = розв'язано, 1 = не розпарсив, 2 = лишились конфлікти, 255 = бінарний (fallback до версії target)
* Формула mergiraf підтверджена у homebrew-core: версія 0.17.0, bottled
* Env-змінні: `GETW_MERGE_MODEL` (модель claude, default `sonnet`), `GETW_MERGE_CURSOR_MODEL` (модель cursor-agent), `GETW_NO_MERGIRAF=1` (вимкнути Tier 2)
* Агент: `claude -p "$prompt" --permission-mode acceptEdits --allowedTools "Edit,Write,MultiEdit,Read" --model "${GETW_MERGE_MODEL:-sonnet}"`, фолбек `cursor-agent -p --force --model "${GETW_MERGE_CURSOR_MODEL:-claude-4.6-sonnet-medium}"`
* Функціональні тести: `/tmp/getw_func/run.sh` (всі 8 merge-кейсів у тимчасовому git-репо), `/tmp/getw_bun/run.sh` (bun.lock defer+regen з fake-bun), `/tmp/getw_t2/run.sh` (fake-mergiraf + реальний `git merge-file --diff3`)
* Перевірка синтаксису: `zsh -n` на витягнутому `ZSH_SCRIPT` через `node --input-type=module`
* `git apply --3way` відкинуто через `does not match index` при двосторонніх змінах тих самих tooling-файлів (`CLAUDE.md`, `.n-cursor.json`, `bun.lock`, `.cursor/rules/*`)
* Обговорення `git rerere` як потенційного Tier (-1): відкинуто в поточній ітерації через складність емуляції unmerged index-записів поза реальним git merge/rebase
* Change-файли додано в `npm/.changes/` через `npx @nitra/cursor change` з bump `minor` (Added) та `patch` (Fixed, Changed)

## Update 2026-06-03

Після успішного мержу скрипт виконує `git reset` (без параметрів), щоб усі прикладені зміни лишались unstaged — індекс не змінюється. Це додатковий крок, необхідний через те, що `git merge-file` не інтегрований у git index і сам по собі не стейджить результат.

## Update 2026-06-03

Реалізація детермінованого вердикту в `_getw_files_with_markers`: функція викликається після роботи агента як `local leftover=$(_getw_files_with_markers "$conflict_files")`; якщо результат непорожній — скрипт повертає `return 1` зі збереженням worktree і повідомляє користувача. Агент не видаляє жодних файлів — вердикт повністю у скрипті.
