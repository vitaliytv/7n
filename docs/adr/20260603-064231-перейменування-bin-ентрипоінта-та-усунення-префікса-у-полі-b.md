## ADR Перейменування bin-ентрипоінта та усунення префікса `./` у полі `bin`

## Context and Problem Statement
`npm publish` видавав попередження `"bin[7]" script name bin/7.js was invalid and removed` і мовчки видаляв весь `bin`-запис з опублікованого пакету, що унеможливлювало встановлення команди через `npm i -g`. Початкова гіпотеза — цифрове ім'я команди `7` — виявилася хибною.

## Considered Options
* Залишити `bin/7.js` і змінити лише ключ команди (`7` → `7n`, `n7` тощо)
* Перейменувати `bin/7.js` → `bin/cli.js` (запит користувача) і виправити значення `bin`
* Залишити `./` у значенні `bin` після перейменування файлу

## Decision Outcome
Chosen option: "Перейменувати `bin/7.js` → `bin/cli.js` та задати значення `bin` без префікса `./`", because емпіричне тестування (`npm publish --dry-run` у temp-пакетах) довело, що саме префікс `./` у значенні `bin` є причиною видалення запису нормалізатором npm під час `publish` — незалежно від імені ключа чи назви файлу. Запит на перейменування файлу надійшов від користувача.

### Consequences
* Good, because `npm publish --dry-run` перестав видавати `auto-corrected` / `invalid and removed` — запис `bin` тепер потрапляє в опублікований пакет.
* Bad, because `npm pkg fix` не виявляє цю проблему і залишає `./` як є, тому порада npm у попередженні («run npm pkg fix») є марною і може ввести в оману.

## More Information
- Файл: `npm/package.json`, поле `bin`: `{ "cli": "bin/cli.js" }` (без `./`)
- Поле `scripts.start` також оновлено: `"bun ./bin/cli.js"` (у `start` `./` залишається — там нормалізатор npm не застосовується)
- Перевірено командою: `npm publish --dry-run 2>&1 | grep -iE "invalid|removed|auto-corrected"`
- Ключові тести в temp-директорії: `"./bin/cli.js"` → ❌ removed; `"bin/cli.js"` → ✅ clean

## ADR Назва пакету `@7n/n` замість `@7n/7`

## Context and Problem Statement
Реальна публікація `npm publish` повертала `E400 Bad Request – That word is not allowed` для пакету `@7n/7`. Назва `7` в npm знаходиться в серверному денлисті і не може бути обійдена локальними фіксами.

## Considered Options
* `@7n/7` — оригінальна назва, заблокована денлистом
* `@7n/cli` — відповідає назві команди та файлу `bin/cli.js`; перевірена `npm view` — вільна
* `@7n/n` — мінімальна, однолітерна назва; узгоджується зі скоупом; перевірена `npm view` — вільна
* `@vitaliytv/n7` — під власним скоупом без потреби в організації

## Decision Outcome
Chosen option: "`@7n/n`", because користувач хотів зберегти цифровий характер у назві (питав «а не можна саме цифру?»), цифра `7` заборонена денлистом, а однолітерний варіант `n` у скоупі `@7n` є найближчим допустимим компромісом. Скоуп `@7n` валідний — організація `7n` належить користувачеві.

### Consequences
* Good, because `npm publish --dry-run` з назвою `@7n/n` проходить чисто з `public access`; скоуп `@7n` не потребує додаткового налаштування.
* Bad, because денлист npm перевіряється лише на сервері під час реального `PUT` — dry-run не гарантує прийнятність назви; транскрипт не містить підтвердження успішної реальної публікації.

## More Information
- Перевірка доступності: `npm view @7n/n version` → `E404 Not Found` (вільне)
- Перевірка доступності: `npm view @7n/7 version` → `E404 Not Found` (вільне технічно, але заблоковане денлистом при PUT)
- Проміжний варіант `@7n/cli` теж пройшов dry-run, але замінений на `@7n/n` за запитом користувача
- Файл: `npm/package.json`, поле `name`

## ADR Додавання `publishConfig.access: "public"` для scoped-пакету

## Context and Problem Statement
Реальна публікація `npm publish` повертала `E402 Payment Required – You must sign up for private packages`. Scoped-пакети (`@scope/...`) npm за замовчуванням трактує як приватні, що вимагає платного плану.

## Considered Options
* Публікувати з флагом `--access public` щоразу вручну
* Додати `publishConfig.access: "public"` у `package.json` (постійно)

## Decision Outcome
Chosen option: "Додати `publishConfig.access: \"public\"` у `package.json`", because це закріплює поведінку в репо назавжди — ані розробник, ані CI не мають пам'ятати про флаг. `npm publish --dry-run` підтвердив: `Publishing to https://registry.npmjs.org/ with tag latest and public access`.

### Consequences
* Good, because transcript фіксує очікувану користь: `npm publish` без додаткових аргументів завжди публікує як public.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Файл: `npm/package.json`, поле `"publishConfig": { "access": "public" }`
- Перевірено: `npm pkg get publishConfig` → `{ "@7n/n": { "access": "public" } }`
- Альтернативна разова команда: `npm publish --access public`
