## ADR Виокремлення ядра delta-мерджу в `merge.js` та нова команда `getpull`

## Context and Problem Statement
Командa `getw` у `npm/getw.js` містила вбудований механізм «інтелектуального» багатокрокового мерджу через zsh. Потрібно було додати нову команду `getpull`, що виконує той самий мердж, але джерелом є `origin/<гілка>`, а не локальний git-worktree. Дублювати zsh-логіку в обох файлах неприпустимо.

## Considered Options
* Винести спільну zsh-логіку в окремий JS-модуль `merge.js` (константа `MERGE_ZSH_LIB` + хелпер `runZsh`), яку імпортують і `getw.js`, і `getpull.js`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Винести спільну zsh-логіку в `merge.js`", because повторне використання без дублювання zsh-коду через спільну константу `MERGE_ZSH_LIB` дозволяє обом командам (`getw`, `getpull`) користуватися тим самим набором тир-резолверів і логікою `_n7merge_delta`.

### Consequences
* Good, because transcript фіксує очікувану користь: обидві команди пройшли `zsh -n` і `vitest run` (28/28) з одного й того ж ядра — без дублювання zsh-фрагментів.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `npm/merge.js` — новий модуль; експортує `MERGE_ZSH_LIB` (рядок-фрагмент zsh із функцією `_n7merge_delta`) та `runZsh(script, spawnFn, argv)`.
- `npm/getw.js` — рефакторинг: замість вбудованого zsh-коду імпортує `{ MERGE_ZSH_LIB, runZsh }` з `./merge.js`.

## ADR Delta-підхід (merge-base + git apply) замість повного `git checkout` у `getpull`

## Context and Problem Statement
Перша реалізація `getpull` використовувала `git checkout origin/$branch -- .`, що перезаписує **всі** tracked-файли версією з origin незалежно від незакомічених змін у робочому дереві. Після першого реального запуску `bun ./npm/bin/n.js getpull` користувач запитав, чи не було затерто корисних даних — що виявило потенційно деструктивну природу підходу.

## Considered Options
* Повний `git checkout origin/$branch -- .` — перший варіант реалізації
* Delta-мердж: лише патч `merge-base(HEAD, origin/$branch)..origin/$branch` через `git apply` + багатоступеневі резолвери (той самий алгоритм, що вже використовував новий `getw`)

## Decision Outcome
Chosen option: "Delta-мердж через `_n7merge_delta`", because повний `git checkout` затирає незакомічені зміни tracked-файлів, що є деструктивним на «брудному» дереві; delta-підхід переносить лише нові зміни відносно спільного предка, зберігаючи локальні правки.

### Consequences
* Good, because transcript фіксує очікувану користь: `zsh -n` нового скрипту `getpull` — OK; 28/28 тестів — OK; зміни джерела (`origin/<гілка>` замість worktree) ізольовані у `getpull.js`, ядро `_n7merge_delta` у `merge.js` незмінне.
* Bad, because delta-підхід потребує `git fetch` перед мерджем (додатковий мережевий запит); transcript не фіксує це як проблему.

## More Information
- `npm/merge.js` — `MERGE_ZSH_LIB` містить zsh-функцію `_n7merge_delta(ours, source)`: `git diff merge-base(ours,source) source | git apply --index`; після — ітеративна розвʼязка конфліктів через тири (Tier 0–3).
- `npm/getpull.js` — передає `ours=HEAD`, `source=origin/$branch`; перед зверненням до `_n7merge_delta` виконує `git fetch origin $branch`.
- Перша (деструктивна) реалізація — `git checkout origin/$branch -- . && git reset .` — відхилена після user-запиту про безпеку даних.
- Перевірка: `zsh -n` → OK; жоден наявний коміт і жоден untracked-файл не постраждали при реальному запуску першої реалізації лише через те, що `origin/main` був попереду локального HEAD на 3 коміти.
