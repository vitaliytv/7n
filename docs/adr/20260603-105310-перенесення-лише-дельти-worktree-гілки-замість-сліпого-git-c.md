## ADR Перенесення лише дельти worktree-гілки замість сліпого `git checkout <branch> -- .`

## Context and Problem Statement
Команда `getw` переносила файли з worktree-гілки у поточну гілку через `git checkout "$target_branch" -- .`, що замінювало **всі** файли, включно з тими, що змінювалися виключно в поточній гілці. Файли, не зачеплені у worktree, але оновлені в `main`, затирались їхньою старою версією без жодного попередження про конфлікт.

## Considered Options
* Залишити `git checkout <branch> -- .` (поточна поведінка)
* Переносити лише дельту `merge-base..target` через `git diff --binary | git apply`

## Decision Outcome
Chosen option: "Переносити лише дельту `merge-base..target`", because це єдиний спосіб не зачіпати файли, що змінювались виключно в поточній гілці: `git diff "$merge_base" "$target_branch"` містить лише те, що справді змінила worktree-гілка, а `git apply` без `--index` кладе результат як unstaged, не зачіпаючи файли поза патчем.

### Consequences
* Good, because файли, змінені тільки в поточній гілці, більше не перезаписуються старою версією з worktree.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/getw.js`; ключові команди: `git merge-base`, `git diff --binary "$merge_base" "$target_branch"`, `git apply --whitespace=nowarn`.

## ADR Пофайловий 3-way merge через `git merge-file` замість `git apply --3way` або `git apply --reject`

## Context and Problem Statement
Перша реалізація конфліктного шляху використовувала `git apply --3way`, що падало з помилкою `does not match index` — обидві гілки незалежно правили ті самі tooling-файли, тож preimage-блоби в патчі не збігалися з індексом `main`. Друга спроба (`git apply --reject`) делегувала видалення `.rej`-файлів LLM-агенту, що крихко: агент міг не мати дозволу на `rm`, міг забути видалити, а скрипт без перевірки «чи справді видалено» не мав детермінованого вердикту про успіх.

## Considered Options
* `git apply --3way`
* `git apply --reject` + делегування видалення `.rej` агентові
* Пофайловий `git merge-file --diff3` (без індексу) з маркерами у файлах; вердикт — grep маркерів скриптом

## Decision Outcome
Chosen option: "Пофайловий `git merge-file --diff3`", because він працює лише по файловій системі (без індексу), тож помилки `does not match index` неможливі в принципі; є неатомарним по файлах; скрипт сам ставить маркери й сам грепає залишок — вердикт детермінований і не залежить від дій агента.

### Consequences
* Good, because скрипт є єдиним власником вердикту про успіх: агент лише прибирає маркери у вже підготовлених файлах (дозволи звужено до `Edit,Write,MultiEdit,Read`).
* Good, because `--diff3` лишає base-секцію `|||||||`, яку потребує mergiraf для структурного авторезолву.
* Bad, because `git merge-file` маркує конфлікт навіть для сусідніх (не лише перетинних) змін — це стандартна поведінка diff3, такі «зайві» конфлікти теж підуть до агента.

## More Information
Файл `npm/getw.js`; helper `_getw_files_with_markers` грепає `^(<<<<<<<|>>>>>>>)`; функціональний тест усіх кейсів у `/tmp/getw_func/run.sh`.

## ADR Багаторівневий пайплайн авторезолву конфліктів: `git merge-file` → mergiraf → LLM-агент

## Context and Problem Statement
Після впровадження `git merge-file` залишок із конфліктними маркерами одразу передавався LLM-агенту. Агент — найдорожчий і найповільніший інструмент; частина «конфліктів» від line-based diff3 є хибними (зміни у різних частинах AST) і можуть бути розв'язані детерміновано.

## Considered Options
* Одразу LLM-агент на все, що залишилось після `git merge-file`
* Mergiraf (структурний AST-авторезолвер) як проміжний tier перед агентом
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Mergiraf як Tier 2 між `git merge-file` і LLM-агентом", because він розв'язує конфлікти, які line-based diff3 позначив безпідставно (зміни у різних вузлах дерева), і підтримує 25+ мов через tree-sitter. Агент (Tier 3) отримує лише справжній залишок. Вердикт усе одно за скриптом (grep маркерів), тож tier 2 не може «схвалити» незавершений мерж.

### Consequences
* Good, because transcript фіксує очікувану користь: агент залучається лише на справжні семантичні конфлікти, яких детерміновані tier'и не розв'язали.
* Good, because mergiraf авто-встановлюється через `brew install mergiraf` (фолбек `cargo install --locked mergiraf`); вимикається через `GETW_NO_MERGIRAF=1`.
* Bad, because mergiraf не обробляє незнайомі типи (`.mdc`, `.md`, lock-файли) — лишає маркери й вони падають до агента, що є очікуваною безпечною поведінкою.

## More Information
Файли `npm/getw.js`, `npm/README.md`; helper `_getw_ensure_mergiraf` (brew → cargo), `_getw_mergiraf_solve`; Tier-порядок: Tier 0 `git apply`, Tier 1 `git merge-file --diff3`, Tier 2 `mergiraf solve`, Tier 3 LLM-агент. Env-змінні: `GETW_MERGE_MODEL`, `GETW_MERGE_CURSOR_MODEL`.

## ADR Окрема обробка `bun.lock`: пропуск merge + умовний `bun install`

## Context and Problem Statement
`bun.lock` як бінарний lock-файл давав шум при пофайловому `git merge-file`; результат merge не має сенсу — lock має бути узгоджений з `package.json` після злиття залежностей. Попередня реалізація запускала `bun install` завжди, коли `bun.lock` був у дельті, — навіть якщо обидва lock-и ідентичні.

## Considered Options
* Мержити `bun.lock` як звичайний файл через merge-file/mergiraf/агент
* Завжди брати версію `target` (worktree)
* Пропускати merge + запускати `bun install` лише коли lock-и реально відрізняються

## Decision Outcome
Chosen option: "Пропускати merge + запускати `bun install` лише коли lock-и реально відрізняються", because `bun install` є єдиним коректним способом узгодити lock зі змерженим `package.json`; запускати його, коли lock-и вже збігаються — зайва операція. Helper `_getw_bun_lock_differs` порівнює вміст через `cmp` і повертає 0 тільки при розбіжності.

### Consequences
* Good, because `bun install` не запускається даремно, якщо обидва lock-и вже збігаються.
* Good, because lock-файли `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock` теж виключені з merge (беруться з target), хоча для них `bun install` не викликається.
* Bad, because якщо `package.json` змінився, але `bun.lock` не потрапив у дельту (теоретично можливий розрив), `bun install` не запуститься автоматично.

## More Information
Файл `npm/getw.js`; helper `_getw_bun_lock_differs` (порівняння ours з робочого дерева або HEAD vs theirs з `target_branch` через `cmp -s`); змінні `regen_bun`, `regen_bun_path`; `bun install` викликається після повністю успішного мержу (без маркерів).
