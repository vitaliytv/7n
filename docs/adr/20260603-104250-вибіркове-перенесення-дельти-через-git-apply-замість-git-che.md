## ADR Вибіркове перенесення дельти через `git apply` замість `git checkout <branch> -- .` у `getw`

## Context and Problem Statement
Команда `getw` переносила зміни з git-worktree у поточну гілку через `git checkout "$target_branch" -- .`, яка сліпо замінювала **всі** файли робочого дерева вмістом worktree-гілки. Файли, змінені тільки в поточній гілці після відгалуження, перезаписувалися старими версіями з worktree — без попередження про конфлікт.

## Considered Options
* `git checkout "$target_branch" -- .` (поточна поведінка)
* `git apply` на дельту `merge-base..target` (`git diff --binary "$merge_base" "$target_branch" | git apply`)

## Decision Outcome
Chosen option: "`git apply` на дельту `merge-base..target`", because він переносить лише файли, реально змінені у worktree-гілці (відносно спільного предка), не зачіпаючи файли, яких worktree не чіпав, і безпечно сигналізує про конфлікт замість мовчазного затирання.

### Consequences
* Good, because файли, змінені лише в поточній гілці, більше не затираються — ключове виправлення регресії даних.
* Bad, because `git apply` без `--index` атомарний: один файл, що не застосовується (наприклад `bun.lock`), блокує весь патч — транскрипт зафіксував це як проблему, вирішену у наступних рішеннях.

## More Information
Файл `npm/getw.js`, рядок `git diff --binary "$merge_base" "$target_branch" > "$patch_file"` з подальшим `git apply --whitespace=nowarn "$patch_file"`. Виправлення також задокументовано у `npm/README.md` і change-файлі з bump `patch`, секція `Fixed`.

## ADR Пофайловий `git merge-file --diff3` замість `git apply --3way` або `--reject` для обробки конфліктів у `getw`

## Context and Problem Statement
Коли `git apply` не міг накласти патч без конфліктів, перший підхід — `git apply --3way` — падав з помилкою `does not match index`: обидві гілки незалежно змінювали tooling-файли (`.cursor/rules/*`, `.n-cursor.json`, `bun.lock`), тому індекс поточної гілки не збігався з preimage-блобами патча. Другий підхід — `git apply --reject` — делегував виявлення успіху агенту (через видалення `.rej`-файлів), що є ненадійним і залежить від дозволів на `rm`.

## Considered Options
* `git apply --3way` (3-way merge через індекс)
* `git apply --reject` + агент видаляє `.rej`
* Пофайловий `git merge-file --diff3` + агент лише прибирає маркери

## Decision Outcome
Chosen option: "Пофайловий `git merge-file --diff3`", because він працює виключно по файлах без індексу (жодного `does not match index`), є неатомарним (один файл не блокує решту), а детермінований вердикт (grep маркерів `<<<<<<<`/`>>>>>>>`) залишається за скриптом, а не агентом.

### Consequences
* Good, because скрипт сам ставить конфліктні маркери та сам перевіряє їхню відсутність після агента — розподіл відповідальностей чіткий і не залежить від дозволів агента на `rm`.
* Bad, because `git merge-file` може позначати конфліктом **сусідні** (не лише перетинні) зміни — стандартна поведінка diff3, частину таких «зайвих» маркерів вирішує mergiraf у наступному tier'і.

## More Information
Функціональний тест на тимчасовому репозиторії підтвердив усі кейси: `ours_only.txt` збережено, `theirs_only` перенесено, неперетинні зміни злиті без маркерів, додавання/видалення оброблено, справжній конфлікт отримав маркери. Файл `npm/getw.js`, функція `getw()`, блок після `else`. Merge-base обчислюється через `git merge-base "$current_branch" "$target_branch"`. Прапорець `--diff3` необхідний для сумісності з `mergiraf solve`.

## ADR Багаторівневий резолв конфліктів: `git merge-file` → `mergiraf` → LLM-агент у `getw`

## Context and Problem Statement
Після переходу на пофайловий `git merge-file` залишалися «зайві» конфлікти — зміни в різних частинах синтаксичного дерева, які line-based diff3 позначав конфліктом через сусідство рядків. Виклик LLM-агента для всіх таких випадків є надлишковим і повільним.

## Considered Options
* Одразу передавати всі маркери LLM-агенту
* Додати структурний AST-авторезолвер (`mergiraf`) перед агентом

## Decision Outcome
Chosen option: "Додати `mergiraf` як Tier 2 між `git merge-file` і LLM-агентом", because `mergiraf solve` розв'язує структурні конфлікти детерміновано (без мережі, без LLM), зменшуючи кількість файлів, що дістаються агенту.

### Consequences
* Good, because `mergiraf` підтримує 25+ мов (JS/TS, Go, Rust, Python, YAML, TOML, JSON тощо); невідомі типи (`.mdc`, `.md`, lock-файли) → лишає маркери → без шкоди передаються агенту.
* Bad, because вимагає наявності `mergiraf` у `PATH`; відсутність вирішена авто-встановленням (окреме рішення).

## More Information
Тире `GETW_NO_MERGIRAF=1` вимикає Tier 2. Функція `_getw_mergiraf_solve` у `npm/getw.js` (~рядок 44). Перевірено з fake-mergiraf stub: файл, який stub розв'язав, не потрапив до агента; файл із маркерами, який stub не зміг — потрапив. `mergiraf solve` (версія 0.17.0, homebrew-core) потребує diff3-маркерів (`|||||||`), тому `git merge-file` викликається з прапорцем `--diff3`.

## ADR Авто-встановлення `mergiraf` через `brew install mergiraf` з фолбеком на `cargo`

## Context and Problem Statement
`mergiraf` є опційним інструментом Tier 2, відсутнім у більшості середовищ за замовчуванням. Без нього Tier 2 мовчки пропускається, підвищуючи навантаження на LLM-агента.

## Considered Options
* Мовчки пропускати Tier 2 якщо `mergiraf` відсутній
* Авто-встановлювати через `brew install mergiraf` (формула є в homebrew-core, 0.17.0)

## Decision Outcome
Chosen option: "Авто-встановлення через `brew install mergiraf`", because формула підтверджено існує в homebrew-core і є bottled; фолбек через `cargo install --locked mergiraf` забезпечує роботу на середовищах без Homebrew.

### Consequences
* Good, because transcript фіксує очікувану користь: Tier 2 стає активним без ручного втручання користувача.
* Bad, because `brew install` або `cargo build` під час виконання `getw` може зайняти значний час (особливо cargo); скрипт повідомляє про це явно.

## More Information
Функція `_getw_ensure_mergiraf` у `npm/getw.js`. Вимикається через `GETW_NO_MERGIRAF=1`. Схема авто-встановлення аналогічна вже наявному патерну з `fzf` у тому ж файлі (`brew install fzf`).

## ADR Окрема обробка `bun.lock`: пропуск merge, перегенерація через `bun install`

## Context and Problem Statement
`bun.lock` є автогенерованим файлом, злиття якого через `git merge-file` дає лише «шум» (конфлікти у lock-форматі, неможливі для семантично правильного ручного резолву). Потрібно забезпечити, що після успішного переносу залежності лишаються коректними.

## Considered Options
* Merge `bun.lock` так само як інші файли (через `git merge-file` / агент)
* Пропустити `bun.lock` у циклі merge, перегенерувати через `bun install` після успішного переносу

## Decision Outcome
Chosen option: "Пропустити `bun.lock` у циклі, перегенерувати через `bun install`", because `bun install` завжди генерує коректний lock-файл з поточного `package.json`, що є авторитетнішим джерелом, ніж будь-який merge двох lock-версій.

### Consequences
* Good, because transcript фіксує очікувану користь: `bun.lock` генерується детерміновано і не потрапляє ні до mergiraf, ні до агента.
* Bad, because `bun install` вимагає мережевого доступу і встановленого `bun`; скрипт попереджає, якщо `bun` відсутній або `bun install` завершився з помилкою.

## More Information
Прапорець `regen_bun` виставляється до будь-якого apply через `git diff --no-renames --name-only "$merge_base" "$target_branch" | grep -qE '(^|/)bun\.lock$|(^|/)package\.json$'` — це фінальне рішення після обговорення, що `bun install` має запускатись і при чистому `git apply` (не лише у конфліктній гілці). Аналогічно відкладаються `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock` (але без `bun install` — беруться з target). Функціонально перевірено fake-bun stub у `/tmp/getw_bun/run.sh`.

## ADR Хоїстинг `regen_bun` до фази виявлення дельти (до `git apply`)

## Context and Problem Statement
Спочатку `regen_bun=0` оголошувався всередині гілки `else` (конфліктний шлях Tier 1+). Це означало, що `bun install` **не викликався** коли `git apply` накладав патч чисто (Tier 0), навіть якщо дельта містила зміни `bun.lock` або `package.json`.

## Considered Options
* Зберігати `regen_bun` всередині `else`-гілки (тільки для конфліктного шляху)
* Виставляти `regen_bun` з дельти до `if/elif/else` (для всіх гілок)

## Decision Outcome
Chosen option: "Виставляти `regen_bun` з дельти до будь-якого apply", because `bun install` необхідний щоразу, коли дельта чіпає `bun.lock` або `package.json` — незалежно від того, чи apply пройшов чисто або через merge-file.

### Consequences
* Good, because transcript фіксує очікувану користь: `bun install` запускається послідовно у всіх сценаріях переносу залежностей.
* Bad, because Neutral, because transcript не містить підтвердження наслідку (зміна не завершена: редагування `getw.js` заблоковано запитом на дозвіл наприкінці сесії).

## More Information
Додатковий `git diff --no-renames --name-only` виклик до основного `git diff --binary` — один зайвий обхід git об'єктів, але детермінований і дешевий. Зміна обговорена в останньому повідомленні сесії, реалізація не завершена через відсутність дозволу на запис файлу.
