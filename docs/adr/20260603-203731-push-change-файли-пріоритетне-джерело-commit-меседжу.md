# Change-файли як пріоритетне джерело для commit-меседжу в `push`

**Status:** Accepted
**Date:** 2026-06-03

## Context and Problem Statement

Команда `push` генерує commit-меседж через LLM-агента на основі `git diff --cached`. Але `.changes/`-файли вже містять готовий авторський опис суті зміни (секції `Added`/`Changed`/`Fixed` та `bump`), що є чистішим і точнішим сигналом, ніж синтаксичний аналіз diff.

## Considered Options

* Використовувати лише відфільтрований diff як єдине джерело контексту для агента
* Використовувати `.changes/`-файли як пріоритетне джерело, diff — як фолбек (якщо change-файлів немає)

## Decision Outcome

Chosen option: "Change-файли як пріоритетне джерело, diff — фолбек", because користувач явно попросив: «для коміту пріоритетним будуть changes якщо є, і тільки якщо не має то тоді вже аналіз diff файлів».

### Consequences

* Good, because transcript фіксує очікувану користь: change-файли містять авторський опис у стандартному форматі (`bump`, `section`), що дає агенту прямий і однозначний сигнал про суть замість виведення її з diff.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Реалізація в `npm/push.js`: зібрати `git diff --cached --name-only | grep '.changes/'`; якщо список непорожній — вивести їхній вміст у ctx першим блоком із заголовком «Change-файли (пріоритетне джерело):»; далі — відфільтрований diff із заголовком «Diff (додатковий контекст):». Якщо change-файлів немає — тільки diff.
- Промпт агента оновлено: явна інструкція опиратись на change-файли за пріоритетом, diff — для уточнення scope й технічних деталей.
- Тест у `npm/tests/push.test.mjs`: інваріант `change-файли пріоритетне джерело`.
- Change-файл: `npm/.changes/1780506647000-211678.md`.
