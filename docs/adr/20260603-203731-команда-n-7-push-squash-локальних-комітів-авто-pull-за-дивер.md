## ADR Команда `n-7 push`: squash локальних комітів, авто-pull за дивергенції, push без підтвердження

## Context and Problem Statement
Проєкт мав команди для перенесення змін із worktree (`getw`) та накочування дельти з origin (`getpull`), але не мав зворотного шляху: відправки локальних комітів + незакомічених змін на origin одним squash-комітом із LLM-генерованим меседжем.

## Considered Options
* `getpush` / `pushup` / `cmpush` / `push` — варіанти назви
* STOP і рекомендувати `getpull` вручну, якщо origin має нові коміти
* Авто-pull (`_n7merge_delta`) при дивергенції перед squash

## Decision Outcome
Chosen option: команда `n-7 push` (ім'я `push`) із такими параметрами: `git add -A` (усі файли, включно untracked), `git reset --soft origin/<branch>` для squash, автоматичний `_n7merge_delta` замість STOP при дивергенції, вивід subject + файлів у stdout без інтерактивного `[y/N]`, `--no-verify` на коміті, multi-line меседж українською Gitmoji + Monorepo від LLM-агента (`claude -p` → `cursor-agent -p`), because так вказав користувач явно для кожного з цих параметрів.

### Consequences
* Good, because `git reset --soft origin/<branch>` гарантує fast-forward push до наявної гілки; нова гілка отримує `push -u` автоматично.
* Good, because авто-pull через той самий `_n7merge_delta` (спільне ядро з `getpull`/`getw`) усуває ручний крок і зберігає консистентність поведінки.
* Bad, because `--no-verify` пропускає changelog-hook; відповідальність за change-файл лежить на авторі, а не на hook.

## More Information
Файли: `npm/push.js`, `npm/index.js`, `npm/types/index.d.ts`, `npm/tests/push.test.mjs`, `npm/tests/index.test.mjs`, `npm/README.md`, `npm/.changes/1780506647000-211678.md`. Модель агента: env `N7COMMIT_MODEL` → `N7MERGE_MODEL` → `GETW_MERGE_MODEL` → `sonnet`. Команда `git fetch origin <branch>` виконується **завжди** (щоб перевірка дивергенції була актуальною); delta-merge — лише якщо `! git merge-base --is-ancestor origin/<branch> HEAD`.

## ADR Фільтрація шумних шляхів із diff-контексту для `push`

## Context and Problem Statement
LLM-агент отримував повний `git diff --cached`, куди потрапляв вміст ADR-файлів, CHANGELOG, lock-файлів, build-артефактів тощо — шум заглушував суть коміту і ускладнював генерацію точного меседжу.

## Considered Options
* Передавати агенту повний diff без фільтрації
* Виключати вміст шумних шляхів із diff (але лишати їхні імена у `--name-status` і в коміті)

## Decision Outcome
Chosen option: "виключати вміст шумних шляхів із diff", because користувач попросив зменшити шум при визначенні суті; файли при цьому залишаються в коміті, а агент бачить їхні імена через `--name-status` для визначення scope.

### Consequences
* Good, because агент отримує компактний, сфокусований контекст; transcript фіксує очікувану користь: «зменшити шум для визначення суті коміту».
* Good, because `**/docs/**` сам по собі не ловить кореневий `docs/` — тому додано обидва pathspec: `docs/**` і `**/docs/**`; поведінку перевірено через `git diff … | grep -c '^diff --git.*docs/adr/'` → 0.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Дефолтний набір виключень (pathspec `:(exclude)...`): `docs/**`, `**/docs/**` (уся документація, включно з ADR), `**/CHANGELOG.md`, `**/.changes/**`, `*.lock`, `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`, `**/*.d.ts`, `**/*.snap`, `**/__snapshots__/**`, `**/*.min.js`, `**/*.map`, `dist/**`, `build/**`, `coverage/**`. Env-кнопки: `N7COMMIT_NO_DEFAULT_EXCLUDE=1` (вимкнути дефолти), `N7COMMIT_EXCLUDE="..."` (додати свої глоби), `N7COMMIT_MAX_DIFF_LINES` (ліміт рядків, дефолт 1500). У stdout `docs/adr/`-файли не перелічуються поштучно — виводиться лише `📄 docs/adr/: N файл(ів)`. Тести: `npm/tests/push.test.mjs`.

## ADR Change-файли як пріоритетне джерело для генерації commit-меседжу в `push`

## Context and Problem Statement
Агент отримував для генерації меседжу лише відфільтрований diff; але `.changes/`-файли вже містять готовий людський опис суті (секція `Added`/`Changed`/`Fixed` та `bump`), і цей сигнал чистіший і точніший за синтаксичний аналіз diff.

## Considered Options
* Використовувати лише diff як джерело контексту для агента
* Використовувати `.changes/`-файли як пріоритетне джерело, diff — як фолбек (якщо change-файлів немає)

## Decision Outcome
Chosen option: "change-файли як пріоритетне джерело, diff — фолбек", because користувач явно попросив: «для коміту пріоритетним будуть changes якщо є, і тільки якщо не має то тоді вже аналіз diff файлів».

### Consequences
* Good, because transcript фіксує очікувану користь: change-файли містять авторський опис у стандартному форматі (`bump`, `section`), що дає агенту прямий і однозначний сигнал про суть замість виведення її з diff.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Реалізація: в `push.js` зібрати `git diff --cached --name-only | grep '.changes/'`; якщо список непорожній — вивести їхній вміст у ctx першим блоком із заголовком «Change-файли (пріоритетне джерело):»; далі — відфільтрований diff із заголовком «Diff (додатковий контекст):». Якщо change-файлів немає — тільки diff. Промпт агента оновлено: явна інструкція опиратись на change-файли за пріоритетом, diff — для уточнення scope й технічних деталей. Файли: `npm/push.js`, `npm/tests/push.test.mjs` (тест на присутність інваріанта `change-файли пріоритетне джерело`), `npm/.changes/1780506647000-211678.md`.
