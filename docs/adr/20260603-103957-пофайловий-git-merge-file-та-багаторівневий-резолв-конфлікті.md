## ADR Пофайловий git merge-file та багаторівневий резолв конфліктів у getw

## Context and Problem Statement
Команда `getw` в `npm/getw.js` переносила зміни з git-worktree у поточну гілку через `git checkout <branch> -- .`, що **сліпо перезаписувало** робоче дерево цілком — файли, змінені лише в поточній гілці, затиралися старою версією з worktree-гілки без виявлення конфліктів. Потрібен механізм, що переносить **лише дельту** worktree і коректно обробляє справжні конфлікти.

## Considered Options
* `git checkout <branch> -- .` — оригінальний підхід (сліпа заміна)
* `git apply` (чиста дельта merge-base..target)
* `git apply --3way` — потребує індексу; падає з «does not match index» на dirty робочому дереві
* `git apply --reject` — без індексу, але атомарний; потребує делегування видалення `.rej` агентові
* Пофайловий `git merge-file --diff3` + tiered авторезолвери (обраний варіант)

## Decision Outcome
Chosen option: "Пофайловий `git merge-file --diff3` з багаторівневим резолвом конфліктів", because це єдиний підхід, що одночасно: (1) уникає `does not match index` (працює лише по файлах, без індексу); (2) дає детермінований вердикт скрипта (grep маркерів); (3) чітко розподіляє ролі — скрипт ставить маркери й виносить вердикт, агент лише прибирає маркери, і (4) дозволяє дешеві детерміновані tier'и перед дорогим агентом.

Рівні резолву (від найдешевшого):
- **Tier 0** — `git apply` (чистий патч дельти)
- **Tier 1** — `git merge-file --diff3` per-file (без індексу, `--diff3` для сумісності з mergiraf)
- **Tier 2** — `mergiraf solve` (AST/tree-sitter авторезолвер; авто-встановлюється через `brew install mergiraf`, фолбек `cargo`)
- **Tier 3** — LLM-агент (`claude -p --permission-mode acceptEdits`, фолбек `cursor-agent -p`) лише на залишок з маркерами

`bun.lock` обробляється окремо: виключається з merge-циклу (`regen_bun=1`) і перегенеровується через `bun install` після успішного мержу — але **лише** якщо `bun.lock` присутній у дельті worktree-гілки відносно merge-base і `git apply` не спрацював чисто.

### Consequences
* Good, because файли, змінені лише в поточній гілці, більше не затираються — `git merge-file` зберігає «ours» там, де «theirs» не міняло ці рядки.
* Good, because вердикт про успіх мержу — детермінований grep маркерів у скрипті, а не залежить від поведінки агента (уникає сценарію, коли агент «забув» видалити `.rej`).
* Good, because `bun.lock` завжди фінально консистентний зі `package.json` після мержу.
* Bad, because `git merge-file` може генерувати конфліктні маркери навіть для **сусідніх** (не лише перетинних) змін — такі false-conflict передаються Tier-2/3 замість зливання автоматично.
