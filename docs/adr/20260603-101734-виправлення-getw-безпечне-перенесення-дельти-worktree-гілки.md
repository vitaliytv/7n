## ADR Виправлення getw: безпечне перенесення дельти worktree-гілки без затирання поточної гілки

## Context and Problem Statement
Команда `getw` переносила зміни з git-worktree-гілки у поточну гілку через `git checkout "$target_branch" -- .`, що повністю перезаписувало всі файли робочого дерева вмістом worktree-гілки. Файли, змінені лише в поточній гілці (але незаймані у worktree-гілці), затиралися старою версією без будь-якого виявлення конфліктів, і зміни незворотньо губилися.

## Considered Options
* `git checkout "$target_branch" -- .` — сліпе перезаписування всього дерева (вихідна реалізація)
* `git apply` лише дельти (`merge-base..target`) без `--index`
* `git apply --3way` з подальшим LLM-мержем конфліктних маркерів
* `git apply --reject` (неатомарний, без індексу) + LLM-видалення `.rej`-файлів
* Пофайловий `git merge-file` для побудови маркерів, LLM прибирає лише маркери, скрипт виносить вердикт

## Decision Outcome
Chosen option: "Пофайловий `git merge-file` + LLM-агент прибирає маркери", because попередні варіанти мали критичні вади: `git apply --3way` падав з `does not match index` на брудному робочому дереві, де обидві гілки незалежно правили ті самі файли; `git apply --reject` + делегування видалення `.rej` Клоду було крихким — вердикт про успіх залежав від дозволів агента на `rm`, а не від детермінованої перевірки скрипта. `git merge-file` працює лише по файлах без індексу (немає `does not match index`), детерміновано ставить конфліктні маркери, а скрипт самостійно перевіряє їх відсутність через `_getw_files_with_markers` після роботи агента.

### Consequences
* Good, because файли, змінені лише в поточній гілці, більше не затираються — `git merge-file` зберігає їх версію (підтверджено функціональним тестом кейсу «тільки ours змінив»).
* Good, because вердикт про успіх мержу виноситься скриптом детерміновано (перевірка маркерів `<<<<<<<` / `>>>>>>>`), а не залежить від здатності агента видаляти файли.
* Good, because `git merge-file` не торкається git-індексу, тому проблема `does not match index` усунена в принципі.
* Good, because агент (`claude -p --permission-mode acceptEdits`, фолбек `cursor-agent -p`) отримує мінімальні права `Edit,Write,MultiEdit,Read` — лише прибирає маркери у вже існуючих файлах.
* Bad, because `git merge-file` позначає конфліктом навіть **сусідні** (не лише перетинні) зміни, що може породжувати дрібні штучні конфлікти, які все одно йдуть до агента.
* Bad, because для успішного запуску агентського мержу необхідна наявність `claude` або `cursor-agent` у `PATH`; за їх відсутності worktree зберігається, але автоматичного розв'язання немає.

## More Information
* Змінені файли: `npm/getw.js`, `npm/README.md`
* Нові helper-функції в `getw.js`: `_getw_resolve_with_agent`, `_getw_files_with_markers`
* Ключові команди у реалізації: `git merge-file -p -L ... ours base theirs`, `git diff --no-renames --name-only`, `git cat-file -e "$target_branch:$rel"`
* Агентська CLI-конвенція: `claude -p "$prompt" --permission-mode acceptEdits --allowedTools "Edit,Write,MultiEdit,Read" --model "${GETW_MERGE_MODEL:-sonnet}"`, фолбек `cursor-agent -p --force --model "${GETW_MERGE_CURSOR_MODEL:-claude-4.6-sonnet-medium}"`
* Функціональний тест всіх кейсів merge-логіки проведено в `/tmp/getw_func/run.sh` на тимчасовому git-репозиторії
* Change-файли (bump `patch`/`minor`, секції `Fixed`/`Added`/`Changed`) додано через `npx @nitra/cursor change`; `npx @nitra/cursor fix changelog` — exit 0

## ADR Стратегія обробки конфліктів у getw: скрипт виносить вердикт, LLM лише усуває маркери

## Context and Problem Statement
Після впровадження `git apply --reject` вердикт про успіх мержу фактично делегувався LLM-агенту: критерієм завершення вважалась відсутність `.rej`-файлів, які мав видаляти сам агент. Це крихко — залежить від дозволів агента на `rm`; агент може їх не видалити або видалити передчасно, спричинивши або хибний провал, або мовчазну втрату хунків.

## Considered Options
* Делегувати агенту видалення `.rej`-файлів як сигнал завершення (попередня реалізація)
* Скрипт детерміновано виносить вердикт через перевірку конфліктних маркерів після роботи агента

## Decision Outcome
Chosen option: "Скрипт виносить вердикт через перевірку маркерів", because перевірка `<<<<<<<`/`>>>>>>>` у файлах — детермінована операція скрипта, яка не залежить від дозволів або поведінки агента; агент отримує лише творчу задачу (прибрати маркери) з мінімальними правами, а відповідальність за безпеку даних лишається у скрипті.

### Consequences
* Good, because transcript фіксує очікувану користь: дані не губляться мовчки — якщо маркери лишились, скрипт зберігає worktree і повідомляє користувача (`_getw_files_with_markers`).
* Good, because агенту не потрібні права на `Bash(rm:*)` — достатньо `Edit,Write,MultiEdit,Read`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
* Функція-вердикт: `_getw_files_with_markers` у `npm/getw.js`
* Перевірка після агента: `local leftover=$(_getw_files_with_markers "$conflict_files")` — якщо не порожньо, `return 1` зі збереженням worktree
