## ADR Джерело commit-меседжу в push: LLM-синтез з `.changes/`-контекстом

## Context and Problem Statement
У `npm/push.js` під час `push`-операції завжди викликається LLM (`pi -p` або `claude`) для генерації commit-меседжу. Виникло питання, чи `.changes/*.md`-файли взагалі аналізуються при формуванні меседжу — і яка роль LLM, коли ці файли є.

## Considered Options
* LLM-синтез завжди: `.changes/`-файли — приоритетний контекст для `pi -p`, але LLM-виклик не пропускається
* Bypass LLM: якщо є `.changes/`-файли — меседж будується механічно з їх frontmatter/body, без LLM

## Decision Outcome
Chosen option: "LLM-синтез завжди з `.changes/`-пріоритетом", because `.changes/`-файли подаються як «ПЕРШОДЖЕРЕЛО наміру» у `$ctx`-промпт (рядок 216 `push.js`), а LLM синтезує на їх основі коректний Gitmoji-меседж. Без LLM неможливо гнучко синтезувати scope, emoji і тіло з кількох різнорідних change-файлів.

### Consequences
* Good, because transcript фіксує очікувану користь: LLM завжди отримує пріоритетний опис намірів із `.changes/`, а не «гадає» із diff.
* Bad, because LLM-виклик завжди додає затримку і вартість — навіть коли `.changes/` уже повністю описують зміни і синтез «очевидний».

## More Information
- `npm/push.js:51–70` — виклик `pi -p --no-session --no-context-files --no-tools "$prompt"`
- `npm/push.js:210–224` — збір `changes_list` і формування секції «ПЕРШОДЖЕРЕЛО» у `$ctx`
- `npm/push.js:42` — рядок промпту: «Якщо в контексті є секція «Change-файли» — будуй меседж НАСАМПЕРЕД на їхньому описі»
