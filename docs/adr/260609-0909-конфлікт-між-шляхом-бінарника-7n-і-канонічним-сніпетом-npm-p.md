## ADR Конфлікт між шляхом бінарника `7n` і канонічним сніпетом `npm-publish.yml`

## Context and Problem Statement
Workflow `.github/workflows/npm-publish.yml` у репо `vitaliytv/7n` завершився з помилкою `Cannot find module '/home/runner/work/7n/7n/npm/bin/n-cursor.js'`. Крок `release-publish` викликає `node npm/bin/n-cursor.js release`, тоді як у пакеті `@7n/n` бінарник насправді називається `npm/bin/n.js` (`package.json` → `"bin": { "n": "bin/n.js" }`). Паралельно з'ясувалось, що канонічний сніпет у `nitra/cursor` (`npm/rules/npm-module/policy/npm_publish_yml/template/npm-publish.yml.snippet.yml`) також містить `node npm/bin/n-cursor.js release`, а рушій `template.mjs` звіряє значення поля `run` буквально.

## Considered Options
* Виправити лише `7n/.github/workflows/npm-publish.yml` — замінити `n-cursor.js` на `n.js` і прийняти розбіжність з template-сніпетом (проходитиме lint-перевірку в `nitra/cursor` з порушенням).
* Оновити канонічний сніпет у `nitra/cursor` — змінити шлях в `template/npm-publish.yml.snippet.yml` на той, що відповідає `@7n/n`, але це зламає всі проєкти, де бінарник досі `n-cursor.js` (зокрема сам `nitra/cursor`).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "дослідження тривало без прийнятого рішення", because transcript завершився в точці діагностики — знайдено конфлікт, але явного вибору стратегії не зафіксовано.

### Consequences
* Good, because transcript фіксує очікувану користь: діагностика точно визначила два різних бінарники (`npm/bin/n.js` у `@7n/n` vs `npm/bin/n-cursor.js` у `@nitra/cursor`) і те, що правка в одному репо не автоматично виправляє інше.
* Bad, because рушій `template.mjs` порівнює поле `run` буквально, тому будь-яке виправлення `7n` без синхронного оновлення сніпета спричинить lint-порушення `npm_publish_yml`; правка сніпета без параметризації, натомість, зламає `nitra/cursor`'s власний workflow.
