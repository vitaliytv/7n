## ADR Вибіркове перенесення дельти через `git apply` замість `git checkout -- .`

## Context and Problem Statement
Команда `getw` використовувала `git checkout "$target_branch" -- .` для перенесення змін з worktree-гілки у поточну гілку. Це сліпо перезаписувало всі файли робочого дерева версіями з target-гілки, затираючи зміни, зроблені тільки у поточній гілці (бо в target лежали їх старі копії). Конфлікти при цьому не виявлялися.

## Considered Options
* `git checkout "$target_branch" -- .` — попередній підхід
* `git diff --binary "$merge_base" "$target_branch" | git apply` — лише дельта від спільного предка

## Decision Outcome
Chosen option: "git diff | git apply", because перенесення дельти (від `merge_base` до target) без `--index` чіпає лише файли, реально змінені у worktree-гілці; файли, змінені тільки у поточній гілці, лишаються недоторканими.

### Consequences
* Good, because зміни поточної гілки більше не затираються версіями з target.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл `npm/getw.js`, зміна рядка `git checkout "$target_branch" -- .` → `git diff --binary "$merge_base" "$target_branch" | git apply --whitespace=nowarn`. Додано helper `git merge-base "$current_branch" "$target_branch"` для обчислення спільного предка.

## ADR Пофайловий `git merge-file --diff3` замість `git apply --3way` для конфліктного мержу

## Context and Problem Statement
Перша спроба обробляти конфлікти через `git apply --3way` падала з помилкою `does not match index`, тому що обидві гілки (`main` і `main-lint`) незалежно правили ті самі tooling-файли (`.cursor/rules/*`, `CLAUDE.md`, `bun.lock`), і індексний запис у поточній гілці не збігався з preimage патча. Крім того, `git apply` атомарний — один невдалий файл блокує решту.

## Considered Options
* `git apply --3way` — тризначний мерж через індекс
* `git merge-file --diff3` — пофайловий тризначний мерж лише по файлах, без індексу

## Decision Outcome
Chosen option: "git merge-file --diff3", because робота виключно по файлах (без індексу) усуває `does not match index`; `--diff3` залишає base-секцію `|||||||`, яку вміє читати mergiraf (Tier 2); неатомарний цикл по файлах не блокує решту через один збій.

### Consequences
* Good, because transcript фіксує очікувану користь: конфлікт більше не падає на брудному робочому дереві; кожен файл обробляється незалежно.
* Bad, because `git merge-file` позначає конфліктом навіть сусідні (не тільки перетинні) зміни — додаткові маркери, які доведеться розв'язувати вручну або агентом.

## More Information
Функціональний тест у `/tmp/getw_func/run.sh` покрив кейси: тільки theirs, тільки ours, обидва без перетину, обидва з конфліктом, додавання, видалення, binary (`exit 255`). Файл `npm/getw.js`.

## ADR Скрипт як детермінований арбітр успіху мержу (маркери, не агент)

## Context and Problem Statement
В попередній реалізації сигналом успіху мержу слугувало те, що LLM-агент **видалив** `.rej`-файли. Це залежало від дозволу агенту на `rm`, і агент міг просто не видалити файл, або скрипт міг сліпо видалити `.rej` «по факту запуску» — що призводило б до мовчазної втрати незрезолвлених хунків. Користувач явно вказав: вердикт не можна делегувати агенту.

## Considered Options
* Агент видаляє `.rej`-файли — сигнал успіху
* Скрипт грепає конфліктні маркери (`^<<<<<<<|^>>>>>>>`) після агента — детермінований вердикт

## Decision Outcome
Chosen option: "Скрипт грепає конфліктні маркери", because скрипт є єдиним детермінованим арбітром: після того як агент відредагував файли, `_getw_files_with_markers` перевіряє залишки маркерів; агент лише «прибирає маркери» і не отримує повноважень на `rm`.

### Consequences
* Good, because хибнонегативні результати (агент не розв'язав — worktree зберігається) безпечніші за хибнопозитивні (агент не видалив — мерж вважається успішним).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Helper `_getw_files_with_markers` у `npm/getw.js`. Дозволи агенту звужені до `Edit,Write,MultiEdit,Read` — `Bash(rm:*)` видалено.

## ADR Багаторівневий резолв конфліктів: `git merge-file` → mergiraf (авто-встановлення) → LLM-агент

## Context and Problem Statement
Користувач запитав, чи існують детерміновані авторезолвери, які можна поставити перед LLM-агентом, щоб агент отримував лише ті конфлікти, які справді потребують семантичного розуміння.

## Considered Options
* Лише LLM-агент для всіх конфліктів
* Багаторівневий pipeline: `git merge-file` (Tier 1) → `mergiraf solve` (Tier 2) → LLM-агент (Tier 3)

## Decision Outcome
Chosen option: "Багаторівневий pipeline", because кожен рівень дешевший за наступний; `mergiraf` (AST-аналіз через tree-sitter) знімає «помилкові» конфлікти від line-based diff3 без залучення LLM; агент отримує лише справжній залишок.

### Consequences
* Good, because transcript фіксує очікувану користь: mergiraf розв'язує структурні конфлікти автоматично; агент викликається рідше.
* Bad, because компіляція `mergiraf` через `cargo` може зайняти кілька хвилин, якщо `brew` недоступний.

## More Information
Tier 0: чистий `git apply`. Tier 1: `git merge-file --diff3`. Tier 2: `mergiraf solve` — авто-встановлення через `brew install mergiraf`, фолбек `cargo install --locked mergiraf`; вимикається `GETW_NO_MERGIRAF=1`. Tier 3: `claude -p --permission-mode acceptEdits`, фолбек `cursor-agent -p`. Функціональний тест з fake-mergiraf (`/tmp/getw_t2/run.sh`). Формула mergiraf підтверджена у homebrew-core (0.17.0). Файл `npm/getw.js`.

## ADR `bun.lock` не мержити, а перегенеровувати через `bun install`

## Context and Problem Statement
`bun.lock` є бінарно-конфліктним lockfile-файлом, для якого пофайловий `git merge-file` дає лише шум маркерів; mergiraf і агент також не здатні коректно злити lockfile семантично. Потрібна окрема стратегія.

## Considered Options
* Брати версію target (попередній підхід для всіх lockfile)
* Відкласти `bun.lock`, не мержити, після успішного мержу запустити `bun install`

## Decision Outcome
Chosen option: "Відкласти + bun install", because `bun install` перегенеровує `bun.lock` зі змердженого `package.json`; це єдиний семантично коректний спосіб злити lockfile — через менеджер пакетів, а не через текстовий мерж.

### Consequences
* Good, because transcript фіксує очікувану користь: `bun.lock` завжди консистентний зі змердженим `package.json`; до конфліктного списку не потрапляє.
* Bad, because вимагає мережевого доступу під час `bun install`; якщо `bun` відсутній — виводиться попередження, lockfile лишається локальною версією.

## More Information
Прапорець `regen_bun=1` у циклі merge; виклик `bun install` після успішного мержу. Інші lockfile (`package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`) — беруться з target без регенерації. Функціональний тест: `/tmp/getw_bun/run.sh` з fake-bun. Файл `npm/getw.js`.
