# Видалення `./`-префікса зі значення поля `bin` у `package.json`

**Status:** Accepted
**Date:** 2026-06-03

## Context and Problem Statement

При виконанні `npm publish` реєстр npm автоматично прибирав запис `bin` з `package.json` і виводив попередження `"bin[...] script name ... was invalid and removed"`. Початкова гіпотеза про цифрове ім'я ключа або назву файлу не підтвердилась: ключ `cli` і файл `cli.js` теж видалялись при наявності `./` у значенні.

## Considered Options

* Значення `bin` із префіксом `"./bin/cli.js"` — звичний відносний шлях
* Значення `bin` без префікса `"bin/cli.js"` — відносний шлях без `./`

## Decision Outcome

Chosen option: "значення `bin` без префікса `./`", because емпіричний тест у temp-директоріях довів: будь-який `bin`-запис зі значенням у форматі `./bin/...` npm під час `publish` мовчки видаляє незалежно від імені ключа чи назви файлу; без `./` — запис проходить валідацію.

### Consequences

* Good, because `npm publish --dry-run` виходить чистим без `auto-corrected` / `invalid and removed` попереджень, і `bin/n.js` потрапляє в тарбол.
* Bad, because `npm pkg fix` цю проблему не виявляє і залишає `./` як є — порада npm у попередженні (`run npm pkg fix`) вводить в оману.

## More Information

* Файл: `npm/package.json`, поле `bin`: `{ "n": "bin/n.js" }` (без `./`)
* Поле `scripts.start` може містити `./` — там нормалізатор npm під час `publish` не застосовується.
* Перевірка: `npm publish --dry-run 2>&1 | grep -iE "invalid|removed|auto-corrected"`
* Ключові тести в temp-директорії:
  * `"7n": "./bin/cli.js"` → ❌ removed
  * `"cli": "./bin/cli.js"` → ❌ removed
  * `"7n": "bin/cli.js"` → ✅ clean
  * `"cli": "bin/cli.js"` → ✅ clean
