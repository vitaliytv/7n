## ADR Заміна сліпого перезаписування (`git checkout -- .`) на перенесення дельти через `git diff | git apply`

## Context and Problem Statement
`getw` переносив зміни з worktree-гілки у поточну гілку через `git checkout "$target_branch" -- .`, що бере геть усі файли з target і затирає ними робоче дерево. Файли, змінені лише в поточній гілці, але не в worktree-гілці, перезаписувалися старою версією без виявлення конфліктів.

## Considered Options
* Сліпий `git checkout <branch> -- .` (попередній підхід)
* `git diff --binary <merge_base> <target> | git apply` (лише дельта від спільного предка)

## Decision Outcome
Chosen option: "`git diff --binary <merge_base> <target> | git apply`", because це дозволяє переносити лише файли, реально змінені у worktree-гілці, не зачіпаючи файли, змінені тільки в поточній гілці.

### Consequences
* Good, because файли, змінені лише в поточній гілці, більше не затираються старою версією з worktree.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/getw.js`. Ключовий рядок до виправлення: `git checkout "$target_branch" -- .`. Після: `git diff --binary "$merge_base" "$target_branch" | git apply --whitespace=nowarn`. `git apply` без `--index` кладе зміни як unstaged і при конфлікті падає з помилкою, зберігаючи worktree.

## ADR Багаторівневий резолв конфліктів: `git merge-file --diff3` → mergiraf → LLM-агент

## Context and Problem Statement
`git apply --3way` падав з помилкою `does not match index` коли поточна гілка мала локальні зміни у тих самих файлах, що й worktree-гілка, — через невідповідність індексних blob'ів preimage-у патча. `git apply` атомарний, тож один невдалий файл (`bun.lock`) блокував усе перенесення. Подальший варіант з `--reject` делегував видалення `.rej`-файлів агенту, що робило вердикт про успіх недетермінованим.

## Considered Options
* `git apply --3way` (підхід із конфліктними маркерами через індекс)
* `git apply --reject` + агент видаляє `.rej`
* Пофайловий `git merge-file --diff3` + опційний `mergiraf solve` + LLM-агент лише на залишок

## Decision Outcome
Chosen option: "пофайловий `git merge-file --diff3` + опційний `mergiraf solve` + LLM-агент лише на залишок", because `git merge-file` працює виключно по файлах без індексу (усуває `does not match index`), є неатомарним (один файл не блокує решту), а вердикт про успіх залишається детермінованим у скрипті (grep маркерів), а не делегується агенту.

### Consequences
* Good, because скрипт сам ставить diff3-маркери й сам виносить вердикт — агент лише прибирає маркери у вже наявних файлах, без дозволів на `rm` та ін.
* Good, because `mergiraf solve` (Tier 2) структурно розв'язує хибні конфлікти (AST-рівень, 25+ мов) до виклику LLM, зменшуючи кількість «творчих» задач для агента.
* Bad, because `git merge-file` може позначати конфлікт навіть для сусідніх (не лише перетинних) змін — це звичайна поведінка diff3, дрібні маркери підуть на агента.

## More Information
Файли: `npm/getw.js`, `npm/README.md`. Helper-функції: `_getw_files_with_markers`, `_getw_mergiraf_solve`, `_getw_resolve_with_agent`. Функціональний тест з fake-mergiraf підтвердив усі кейси: тільки theirs, тільки ours (збережено), неперетинні (злито без маркерів), конфлікт (маркери + список), додавання, видалення, видалено в target коли ours змінив. Tier'и: 0 — чистий `git apply`; 1 — `git merge-file --diff3`; 2 — `mergiraf solve` (off через `GETW_NO_MERGIRAF=1`); 3 — LLM-агент.

### Consequences

## ADR Окрема обробка `bun.lock` — пропуск мержу та перегенерація через `bun install`

## Context and Problem Statement
`bun.lock` є автогенерованим lock-файлом. Пофайловий merge (`git merge-file`) для нього генерує лише шум, а злитий lock файл може виявитися некоректним. Правильний спосіб отримати актуальний lock — перегенерувати його з вже змердженого `package.json`.

## Considered Options
* Брати версію `bun.lock` з target (попередній підхід)
* Пропустити `bun.lock` під час мержу та перегенерувати через `bun install` після успішного мержу

## Decision Outcome
Chosen option: "пропустити `bun.lock` під час мержу та перегенерувати через `bun install` після успішного мержу", because це єдиний детермінований спосіб отримати консистентний lock-файл — він синхронізується з `package.json`, який на момент виклику вже містить змерджені залежності.

### Consequences
* Good, because `bun.lock` завжди консистентний зі змердженим `package.json`; мержений вручну lock не може бути некоректним.
* Bad, because потребує мережевого доступу або кешу для `bun install`; якщо `bun` відсутній або `bun install` падає — lock залишається в стані local-ours із попередженням.

## More Information
Файл: `npm/getw.js`. Прапорець: `regen_bun=1` у циклі при виявленні `bun.lock`. Виклик після мержу: `bun install > /dev/null 2>&1`. Аналогічно відкладаються `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock` (ті беруться з target без перегенерації). Функціональний тест з fake-bun підтвердив: bun.lock не потрапляє у список конфліктів, `bun install` викликається після успішного мержу.
