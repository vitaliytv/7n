## ADR Виправлення `getw`: дельта-перенесення замість сліпого перезапису

## Context and Problem Statement

Команда `getw` переносила зміни з вибраного git-worktree у поточну гілку через `git checkout "$target_branch" -- .`, що замінювала **весь** вміст робочого дерева файлами з worktree-гілки. Оскільки worktree-гілка могла відставати від поточної гілки, файли, змінені лише в поточній гілці, безмовно перезаписувалися старими версіями — без виявлення конфліктів і без можливості відновлення.

## Considered Options

* `git checkout "$target_branch" -- .` — сліпе перезаписування всього робочого дерева (попередня реалізація)
* Перенесення лише дельти worktree-гілки через `git diff --binary <merge-base> <target> | git apply --whitespace=nowarn`

## Decision Outcome

Chosen option: "Перенесення лише дельти worktree-гілки через `git diff | git apply`", because команда `git checkout <branch> -- .` затирала файли поточної гілки, не зачеплені worktree-гілкою; новий підхід обчислює спільного предка через `git merge-base`, генерує патч лише реально змінених файлів і безпечно зупиняється при конфлікті замість мовчазної втрати даних.

### Consequences

* Good, because файли, змінені лише в поточній гілці, більше не перезаписуються — `git apply` без `--index` чіпає виключно файли з патча.
* Good, because при конфлікті `git apply` повертає ненульовий код виходу, worktree зберігається для ручного злиття, а зміни поточної гілки лишаються недоторканими.
* Good, because зміни кладуться як unstaged (як і раніше очікує решта workflow).
* Bad, because якщо `git merge-base` не знаходить спільного предка (гілки не мають спільної історії), перенесення повністю скасовується — transcript не містить підтверджених негативних наслідків для типового сценарію.
