## ADR Заміна сліпого `git checkout -- .` на точкове перенесення дельти через `git apply`

## Context and Problem Statement
Команда `getw` переносила файли з worktree-гілки в поточну гілку через `git checkout "$target_branch" -- .`, що замінювало **весь** вміст робочого дерева файлами worktree-гілки. Оскільки worktree-гілка відгалужена від main раніше, файли, змінені лише в main, перезаписувалися старими версіями з worktree — зміни в main тихо зникали.

## Considered Options
* `git checkout <branch> -- .` — повне перезаписування (поточна поведінка)
* `git diff --binary <merge-base> <target> | git apply` — передати лише дельту worktree-гілки відносно спільного предка

## Decision Outcome
Chosen option: "`git diff --binary <merge-base> <target> | git apply`", because це переносить лише файли, яких реально торкалась worktree-гілка; файли, змінені тільки в поточній гілці, не зачіпаються.

### Consequences
* Good, because файли, змінені лише в поточній гілці, більше не затираються.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/getw.js`. Команди: `git merge-base`, `git diff --binary`, `git apply --whitespace=nowarn`. Помилка, що спровокувала зміну: тихе перезаписування незакомічених змін `main` при виконанні `getw`.

## ADR Інтелектуальний LLM-мерж замість падіння при конфлікті

## Context and Problem Statement
Після заміни `git checkout` на `git apply` при перетинних змінах у обох гілках `git apply` падав із помилкою і зупиняв перенесення. Користувач попросив, щоб замість падіння конфлікти розв'язував LLM-агент.

## Considered Options
* Падати при конфлікті (зберігати worktree для ручного доведення)
* Викликати `claude -p` / `cursor-agent -p` для розв'язання конфліктних маркерів

## Decision Outcome
Chosen option: "Викликати LLM-агент", because користувач явно попросив інтелектуальний мерж замість падіння; fallback-ланцюг `claude -p` → `cursor-agent -p` відповідає конвенції ADR-хуків проєкту.

### Consequences
* Good, because конфлікти розв'язуються автоматично без ручного втручання.
* Bad, because потребує авторизованого CLI (`claude` або `cursor-agent`) у `PATH`.

## More Information
Файл: `npm/getw.js`. CLI-прапорці: `--permission-mode acceptEdits`, `--model ${GETW_MERGE_MODEL:-sonnet}`. Фолбек: `cursor-agent -p --force --model ${GETW_MERGE_CURSOR_MODEL:-claude-4.6-sonnet-medium}`.

## ADR Скрипт, а не агент, виносить вердикт про успіх мержу (маркери vs `.rej`)

## Context and Problem Statement
Перший варіант з `git apply --reject` делегував агентові видалення `.rej`-файлів як сигнал завершення. Це крихко: залежить від дозволу `rm`, агент може забути видалити файл, а якщо скрипт сліпо видаляв `.rej` після агента — пропущений хунк зникав беззвучно.

## Considered Options
* `git apply --reject` + агент видаляє `.rej` (наявний стан)
* Пофайловий `git merge-file --diff3` (без індексу) — скрипт ставить маркери, агент прибирає маркери, скрипт грепає `^<<<<<<<` / `^>>>>>>>` і виносить вердикт

## Decision Outcome
Chosen option: "`git merge-file --diff3` + grep-вердикт скрипта", because скрипт повинен детерміновано контролювати результат; агент виконує лише творчу частину (прибирає маркери в уже наявних файлах), а рішення «успіх/fail» залишається за grep-перевіркою в скрипті.

### Consequences
* Good, because вердикт детермінований і не залежить від дозволів агента на `rm`.
* Good, because `git merge-file` без `--index` усуває помилку `does not match index`, яку давав `git apply --3way` на dirty-дереві.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `npm/getw.js`. Функції: `_getw_files_with_markers`, `_getw_resolve_with_agent`. Перевірено функціонально на тимчасовому репозиторії (7 кейсів: ours-only, theirs-only, clean merge, conflict, add, delete, binary).

## ADR Багаторівневий ланцюг авторезолверів перед LLM-агентом

## Context and Problem Statement
Користувач запитав, чи є детерміновані авторезолвери, які варто запускати перед дорогим LLM-агентом. Ціль — мінімізувати кількість конфліктів, що доходять до агента.

## Considered Options
* Одразу LLM-агент на всі конфлікти
* Багаторівневий ланцюг: Tier 0 (clean git apply) → Tier 1 (git merge-file --diff3) → Tier 2 (mergiraf solve, AST-авторезолвер) → Tier 3 (LLM-агент)

## Decision Outcome
Chosen option: "Багаторівневий ланцюг з mergiraf як Tier 2", because mergiraf виконує структурний AST-merge через tree-sitter і розв'язує конфлікти, які line-based diff3 позначає хибно; агент отримує лише справжній залишок. Mergiraf опційний (активується якщо в `PATH`, вимикається `GETW_NO_MERGIRAF=1`).

### Consequences
* Good, because зменшується кількість звернень до LLM-агента і відповідна вартість.
* Good, because mergiraf потребує `--diff3`-маркери, які `git merge-file --diff3` вже генерує.
* Bad, because Tier 2 відсутній якщо mergiraf не встановлено і `brew`/`cargo` недоступні.

## More Information
Файл: `npm/getw.js`. Функції: `_getw_ensure_mergiraf`, `_getw_mergiraf_solve`. Auto-install: `brew install mergiraf`, фолбек `cargo install --locked mergiraf`. Перевірено з fake-mergiraf-stub на тимчасовому репо. mergiraf CLI: `mergiraf solve <file>` (exit 0 — розв'язано, 1 — не розпарсив, 2 — лишились маркери).

## ADR Окрема обробка `bun.lock`: пропуск мержу та перегенерація через `bun install`

## Context and Problem Statement
`git merge-file` на `bun.lock` і `package.json` дає неінформативні конфлікти (lockfile генерується детерміновано з `package.json`). Правильна стратегія — пропустити merge і перегенерувати lockfile після злиття.

## Considered Options
* Мержити `bun.lock` через merge-file / mergiraf / агент (наявний стан)
* Пропускати `bun.lock` у циклі merge і запускати `bun install` після успішного мержу
* Запускати `bun install` лише якщо `bun.lock` є в дельті (тільки conflict-шлях)
* Запускати `bun install` якщо `bun.lock` **або** `package.json` є в дельті — незалежно від шляху (clean apply або conflict)

## Decision Outcome
Chosen option: "Детекція за дельтою (`bun.lock` або `package.json`) до розгалуження, `bun install` після `fi`", because користувач явно попросив запускати коли в дельті є `bun.lock` або `package.json`; детекція до розгалуження гарантує однакову поведінку і при чистому `git apply`, і при конфліктному шляху.

### Consequences
* Good, because `bun.lock` завжди консистентний зі змердженим `package.json`.
* Bad, because `bun install` може завантажувати залежності навіть якщо локальна версія вже актуальна.

## More Information
Файл: `npm/getw.js`. Блок детекції: `git diff --no-renames --name-only "$merge_base" "$target_branch" | grep -qE '^(bun\.lock|package\.json)$' && regen_bun=1`. Блок перегенерації: `bun install > /dev/null 2>&1`. Інші lockfiles (`package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`) беруться з target-гілки без перегенерації.
