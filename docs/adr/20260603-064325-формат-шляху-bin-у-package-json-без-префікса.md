## ADR Формат шляху `bin` у `package.json` — без префікса `./`

## Context and Problem Statement

Під час `npm publish` пакета `@7n/n` npm видаляв увесь запис `bin` з попередженням `"script name bin/cli.js was invalid and removed"`. Початково підозрювали, що причина — числова назва команди `7` або назва файлу `7.js`.

## Considered Options

* `"cli": "./bin/cli.js"` — відносний шлях із префіксом `./`
* `"cli": "bin/cli.js"` — відносний шлях без `./`

## Decision Outcome

Chosen option: `"cli": "bin/cli.js"` (без `./`), because емпіричний dry-run показав: будь-який `bin`-запис зі значенням у форматі `"./bin/..."` npm під час `publish` мовчки видаляє; без `./` — запис проходить валідацію навіть для імені `cli`, яке раніше теж відхилялося.

### Consequences

* Good, because `npm publish --dry-run` підтвердив: після прибирання `./` попередження «auto-corrected» зникло, `bin/cli.js` включено в tarball.
* Bad, because поведінка суперечить документації npm (офіційні приклади часто показують `"./bin/..."`), тому розробник без цього ADR не здогадається, чому `npm pkg fix` не виправляє проблему — він зберігає `./`.

## More Information

* `npm/package.json` `bin` field: `"cli": "bin/cli.js"`
* Підтверджено dry-run: `npm publish --dry-run 2>&1 | grep -iE "invalid|removed"` → `(no warnings)` після прибирання `./`
* `npm pkg fix` проблему **не** виявляє — лишає `./` як є.

## ADR Перейменування `bin`-ентрипоінта та команди CLI

## Context and Problem Statement

Файл `npm/bin/n-7.js` перейменовано на `npm/bin/7.js` раніше (до цієї сесії). Під час сесії npm відхиляв запис `bin` через проблеми з форматом шляху та назвою команди, тому ім'я команди та файлу зазнали ітерацій.

## Considered Options

* `"7": "./bin/7.js"` — оригінальне ім'я команди й файлу
* `"7n": "./bin/7.js"` — ім'я команди `7n`, той самий файл
* `"7n": "bin/cli.js"` — ім'я команди `7n`, перейменований файл
* `"cli": "bin/cli.js"` — ім'я команди і файл `cli`

## Decision Outcome

Chosen option: `"cli": "bin/cli.js"`, because користувач явно запросив перейменувати файл у `cli.js` (незалежно від дебагу npm-попередження), а ім'я команди `cli` відповідає назві файлу і не конфліктує з денлистом.

### Consequences

* Good, because transcript фіксує очікувану користь: `npm publish --dry-run` чистий, `bin/cli.js` потрапляє в tarball без попереджень.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

* `npm/bin/cli.js` (раніше `n-7.js`, потім `7.js`): `#!/usr/bin/env node` + `import { run } from '../index.js'`
* `npm/package.json` `scripts.start`: `"bun ./bin/cli.js"` (залишено `./` тут — скрипти `scripts` npm не перевіряє так само, як `bin`)

## ADR Назва npm-пакета: `@7n/n` замість `@7n/7`

## Context and Problem Statement

При спробі `npm publish` пакета з `"name": "@7n/7"` npm повернув `E400 Bad Request: That word is not allowed` — ім'я `7` потрапило в серверний денлист. Перехід через `@7n/cli` і обговорення, чи можна залишити цифру, призвів до вибору `@7n/n`.

## Considered Options

* `@7n/7` — оригінальна назва (числова частина після скоупу)
* `@7n/cli` — збіг з назвою файлу/команди
* `@7n/n` — одна буква `n`, скорочення від `n-cursor` / `7n`

## Decision Outcome

Chosen option: `@7n/n`, because `@7n/7` дав `E400` (денлист), `@7n/cli` прийнятний технічно, але користувач хотів коротшу назву; `@7n/n` вільна на реєстрі (перевірено `npm view @7n/n` → `E404`) і пройшла реальну публікацію без `E400`.

### Consequences

* Good, because transcript фіксує очікувану користь: `+ @7n/n@0.1.0` — пакет опубліковано успішно.
* Bad, because денлист npm перевіряється лише на сервері при реальному `PUT` (dry-run не відтворює `E400`), тому остаточне підтвердження назви можливе лише через публікацію.

## More Information

* `npm/package.json` `name`: `"@7n/n"` (фінальний стан)
* Скоуп `@7n` належить користувачеві (підтверджено скріншотом npmjs.com Account Settings)
* `npm whoami` → `vitaliytv`; `npm view @7n/n version` → `E404` до публікації
* Реальна публікація: `+ @7n/n@0.1.0`

## ADR `publishConfig.access: "public"` для scoped-пакета

## Context and Problem Statement

Після виправлення `bin`-запису `npm publish` повернув `E402 Payment Required: You must sign up for private packages`. Scoped-пакети (`@scope/...`) npm типово публікує як приватні, що потребує платного плану.

## Considered Options

* `npm publish --access public` — прапор при кожній публікації
* `"publishConfig": { "access": "public" }` у `package.json` — постійне налаштування в репо

## Decision Outcome

Chosen option: `"publishConfig": { "access": "public" }` у `package.json`, because зберігає налаштування в репо та усуває необхідність пам'ятати флаг при кожному `publish` або в CI.

### Consequences

* Good, because transcript фіксує очікувану користь: dry-run підтвердив `Publishing to https://registry.npmjs.org/ with tag latest and public access`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

* `npm/package.json`: `"publishConfig": { "access": "public" }`
* `npm pkg get publishConfig` → `{ "@7n/n": { "access": "public" } }`
